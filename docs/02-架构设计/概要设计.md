# SyncAI / 灵悉 AI - 概要设计

**版本：** v1.3  
**更新时间：** 2026-04-30  
**文档定位：** 本文已吸收原 `MVP架构设计.md` 的主干内容，并在 `docs/01-产品/产品需求文档.md`、`docs/02-架构设计/Agent接入设计.md` 之后作为正式概要设计主文档，用于统一后续测试设计、开发拆解、接口细化和实现落地口径。

## 1. 设计目标

本概要设计只服务当前 SyncAI MVP，不扩展到未拍板能力。目标是把三份基线文档收敛成一份可直接执行的工程设计主线，确保后续测试和开发都围绕同一套对象、状态、流程和边界展开。

本文重点回答四件事：
- MVP 到底要实现哪些系统对象和核心流程。
- 用户可见内容、平台内部记录、Codex 接入边界如何区分。
- 串行调度、排队执行、历史回放、搜索、轻量 todo、共享转私有审计如何统一落地。
- 后续详细接口设计、测试用例设计、开发任务拆解应以什么结构继续展开。

---

## 2. 已确认基线约束

以下口径已经拍板，本文只做落实，不再讨论：
- 搜索第一版只覆盖成员消息和 Agent 最终回复。
- 历史回放第一版覆盖消息、状态变化、命令过程摘要，以及 `shared` / `private` 可见性切换节点。
- 第一版默认单管理员执行节点，整个团队共享 1 个管理员本机上的 Codex 执行环境。
- 当前会话运行中时，后续成员消息直接入队，前端显示排队中。
- 共享转私有后，对其他成员隐藏整个会话，但平台保留共享起止时间和当时可见成员范围的审计记录。
- 会话必须在创建成功时立即绑定团队当前 Codex 节点；若节点缺失、不可用或底层会话创建失败，则本次会话创建直接失败，不允许延后到首条消息发送时再绑定。
- 第一阶段主接入只支持 Codex，Claude Code 后续再补。
- MVP 结构固定为团队 > 项目 > 会话。
- Todo 只做轻量 todo：手动从消息提取，字段仅标题和状态，状态仅待办/已完成，保留来源消息跳转，展示在会话侧边栏。

## 3. 设计范围与非目标

### 3.1 本次概要设计覆盖

- 团队、项目、会话三层组织结构。
- 共享会话和私有会话的创建、展示、切换与权限边界。
- 成员消息进入会话后的串行调度、排队执行与状态反馈。
- CodexAdapter 的接入职责、事件归一化和执行回传。
- 消息、事件、todo、审计记录的存储边界。
- 搜索、历史回放、轻量 todo 的数据来源和装配规则。
- 异常情况下的最小恢复约束。

### 3.2 明确不纳入本次实现

- Claude Code 接入实现。
- 多管理员、多执行节点、跨节点迁移或自动故障切换。
- IDE 插件、移动端、平台级多 Agent 编排。
- AI 自动提取 todo、复杂任务系统、复杂审批流。
- 搜索命令过程、文件变化或其他过程事件。
- 历史回放完整镜像终端原始输出。

## 4. 系统上下文与总体边界

### 4.1 角色边界

- 管理员：负责创建团队、维护团队成员、配置并维护团队使用的 Codex 执行节点。
- 普通成员：在项目内查看共享会话、参与共享会话协作、管理自己可见会话中的轻量 todo。
- 平台服务：负责调度、记录、回放、搜索、审计，不直接暴露 Codex 原始运行细节。
- Codex 执行环境：实际完成底层 Agent 会话执行，受 SyncAI 的调度层调用。

### 4.2 系统边界

```text
成员浏览器
  -> Web / WebSocket
SyncAI 应用服务
  -> PostgreSQL（业务数据、消息、事件、todo、审计）
  -> Redis（会话队列、执行锁、实时态缓存）
  -> CodexAdapter（SyncAI 与 Codex 的唯一接入边界）
管理员本机 Codex 执行环境
```

### 4.3 外部依赖边界

- SyncAI 对底层执行能力的依赖在 MVP 阶段只有 Codex。
- 前端和业务服务都不直接依赖 Codex 原始 CLI 输出格式。
- 平台依赖 WebSocket 做实时同步，但消息落库必须先于实时推送成功与否。

### 4.4 推荐部署形态

- 应用服务部署在一台中心服务器。
- PostgreSQL 与 Redis 可与应用服务同机或近邻部署。
- 管理员本机安装并维护 Codex 运行环境。
- MVP 不引入多节点执行集群、统一托管的云端 Agent 池或跨管理员自动故障切换。

## 5. 核心设计原则

- 单会话单底层会话：一个 SyncAI 会话一次只绑定一个 Codex 会话引用，不派生并发副本。
- 串行优先：同一会话在任一时刻只能有一条执行中的消息，其余消息按进入顺序排队。
- 消息与事件分层：用户继续协作依赖消息层，回放与状态跟踪依赖事件层。
- 用户可见内容受产品边界约束：不是所有内部记录都进入搜索或回放。
- 先持久化再展示：成员消息、Agent 最终回复、关键事件、审计记录必须先写入存储，再用于前端展示和后续查询。
- MVP 优先单体应用，不为未来多服务形态提前拆分。
- 只做最小必要恢复：发生错误时保证状态可见、队列不丢、可继续协作，不追求复杂自动容灾。

## 6. 领域对象定义

| 对象 | 含义 | MVP 必要字段 | 说明 |
|---|---|---|---|
| Team | 团队容器 | `id` `name` `created_by` | 顶层协作范围 |
| TeamMember | 团队成员关系 | `team_id` `user_id` `role` | 角色只分管理员/普通成员 |
| Project | 项目容器 | `id` `team_id` `name` `archived_at?` | 用于组织会话，不承担复杂项目管理 |
| Session | 协作会话 | `id` `project_id` `creator_id` `visibility` `runtime_status` `bound_agent_type` `bound_agent_node_id` `bound_agent_session_ref` | MVP 结构核心对象 |
| Message | 会话消息 | `id` `session_id` `sender_type` `sender_id` `content` `processing_status` `is_final_reply` `created_at` | 只把成员消息和 Agent 最终回复作为消息层 |
| SessionEvent | 会话事件 | `id` `session_id` `event_type` `summary` `payload` `created_at` | 回放和状态跟踪使用 |
| Todo | 轻量待办 | `id` `session_id` `source_message_id` `title` `status` `creator_id` | 状态仅待办/已完成 |
| SessionAuditLog | 会话审计记录 | `id` `session_id` `action_type` `previous_visibility` `new_visibility` `visible_scope_snapshot` `shared_started_at` `shared_ended_at` `operator_id` | 共享转私有场景必须保留 |

补充约束：
- `Session.visibility` 只允许 `shared` / `private`。
- `Session.runtime_status` 只允许 `idle` / `queued` / `running` / `completed` / `error`。
- `Message.processing_status` 只允许 `accepted` / `queued` / `running` / `completed` / `failed`。
- `Message.sender_type` 只允许 `member` / `agent`。
- `Todo.status` 只允许 `pending` / `completed`。
- `Session.bound_agent_type` 在 MVP 阶段固定为 `codex`。

## 7. 功能模块拆分

### 7.1 团队与项目模块

职责：
- 创建团队。
- 维护团队成员与角色。
- 在团队下创建项目并组织会话列表。

落地约束：
- MVP 不做复杂项目级权限隔离。
- 默认团队成员都能访问团队下项目中的共享会话。
- 私有会话仍挂在项目下，但仅创建者本人可见。

### 7.2 会话管理模块

职责：
- 创建共享会话和私有会话。
- 保存会话绑定的 Codex 引用。
- 维护会话可见性和运行状态。
- 输出会话详情、会话列表和当前排队状态。

关键约束：
- 创建会话时即建立 SyncAI 会话与底层 Codex 会话的绑定关系。
- 创建成功的前提是已确定团队当前 Codex 节点，并同时写入 `bound_agent_node_id` 与 `bound_agent_session_ref`；绑定失败时不得留下未绑定会话空壳。
- 会话列表展示以 `visibility` 和成员可见范围过滤。
- 共享转私有属于会话层状态切换，不是物理删除。

### 7.3 消息与调度模块

职责：
- 接收成员消息并落库。
- 为会话维护执行锁和 FIFO 队列。
- 在空闲时立即投递到底层执行，在运行中追加到队列。
- 更新消息处理状态和会话运行状态。

关键约束：
- 同一会话同一时刻只允许一条消息处于 `running`。
- 队列顺序必须严格等于消息入队顺序。
- 队列状态变化要通过实时推送和事件记录对前端可见。

### 7.4 Adapter 接入模块

职责：
- 启动或恢复底层 Codex 会话。
- 接收调度层投递的成员消息。
- 监听底层状态变化并归一化为平台事件。
- 提取 Agent 最终回复和命令过程摘要。

关键约束：
- 只有 Adapter 层可以直接感知 Codex 执行细节。
- 业务层消费的是平台统一结构，不直接解析 CLI 临时输出格式。
- 失败时必须返回可读错误摘要，不允许只有黑盒失败。

### 7.5 回放与搜索模块

职责：
- 回放模块按时间轴装配消息、状态变化、命令过程摘要、可见性切换节点。
- 搜索模块只索引和查询成员消息、Agent 最终回复。

关键约束：
- `SessionEvent` 可用于回放，但不进入搜索。
- 命令过程摘要可进入回放，但不进入搜索。
- `session.shared` / `session.privatized` 必须进入回放时间轴，但仍不进入搜索。
- 文件变化、过程日志等内容不在 MVP 搜索范围内。

### 7.6 Todo 模块

职责：
- 从指定消息手动提取轻量 todo。
- 展示会话侧边栏 todo 列表。
- 支持待办/已完成状态切换。
- 支持从 todo 跳回来源消息。

关键约束：
- 不做 AI 自动提取。
- 不做负责人、截止日期、优先级等扩展字段。
- 一个 todo 必须绑定一个 `source_message_id`。

### 7.7 审计与可见性模块

职责：
- 处理共享/私有切换。
- 记录共享起止时间与当时可见成员范围。
- 对会话列表、详情、搜索、回放统一施加可见性过滤。

关键约束：
- 共享转私有后，对其他成员隐藏整个会话。
- 平台必须保留会话曾经共享过的事实。
- 审计记录独立于消息和事件，不因后续可见性变化而删除。

## 8. 数据分层与存储规则

### 8.1 消息层

消息层仅包含两类数据：
- 成员消息。
- Agent 最终回复。

消息层用途：
- 前端主消息流展示。
- 搜索索引来源。
- todo 来源消息跳转。
- 协作上下文续写。

### 8.2 事件层

事件层用于记录结构化过程信息，MVP 至少包括：
- `status.changed`
- `command.summary`
- `message.queued`
- `session.shared`
- `session.privatized`

事件层用途：
- 历史回放。
- 运行状态展示。
- 排队状态反馈。
- 可见性切换留痕。

补充说明：
- Agent 最终回复不写成独立 `SessionEvent`，而是统一写入消息层，并标记 `is_final_reply = true`。

### 8.3 审计层

审计层独立保存共享范围变化事实，至少记录：
- 会话编号。
- 操作类型。
- 共享开始时间。
- 共享结束时间。
- 操作时可见成员范围快照。
- 操作人。

### 8.4 缓存与队列层

Redis 承担：
- 每会话消息队列。
- 每会话执行锁。
- 会话实时状态缓存。

使用原则：
- Redis 只承担运行态和调度态，不替代业务主存。
- 队列中的消息必须在进入 Redis 之前已在 PostgreSQL 持久化。

## 9. 状态模型

### 9.1 会话运行状态

| 状态 | 含义 | 进入条件 | 退出条件 |
|---|---|---|---|
| `idle` | 当前无执行、无待执行消息 | 新建会话或上一轮消费结束后无排队消息 | 有新消息进入 |
| `queued` | 当前有待执行消息但尚未开始执行 | 会话运行中收到新消息，或空闲会话收到消息后进入调度等待 | 某条消息开始执行或队列清空 |
| `running` | 底层 Codex 正在执行 | 调度层拿到执行锁并开始投递 | 当前消息完成或失败 |
| `completed` | 最近一条执行完成 | 收到 Agent 最终回复并完成本轮记录 | 可回落到 `idle` 或继续处理下一条排队消息 |
| `error` | 最近一条执行失败 | Adapter 返回失败或节点不可用 | 人工重试、重新发送后重新进入队列 |

说明：
- `completed` 用于向前端表达“最近一轮已结束”，但不是长期停留态，可在刷新会话聚合状态时回落为 `idle`。
- 如果当前消息完成后队列非空，应直接流转到下一条消息的执行状态，而不是长时间停留在 `completed`。

### 9.2 消息处理状态

| 状态 | 含义 |
|---|---|
| `accepted` | 已接收并落库，等待调度决策 |
| `queued` | 已进入会话队列，等待轮到执行 |
| `running` | 已被投递到底层执行 |
| `completed` | 对应执行已完成，且已有最终回复 |
| `failed` | 对应执行失败，保留错误摘要 |

### 9.3 可见性状态

| 状态 | 含义 |
|---|---|
| `shared` | 当前项目成员可见、可继续参与协作 |
| `private` | 当前仅创建者本人可见 |

说明：
- 可见性影响会话列表、详情、搜索、回放和 todo 侧边栏是否可见。
- 历史审计不受当前可见性反向删除影响。

## 10. 核心时序流程

### 10.1 创建共享会话

```text
成员创建共享会话
  -> SessionService 读取团队当前 Codex 节点
  -> AdapterService 创建或初始化底层 Codex 会话
  -> SessionService 写入 Session(shared, bound_agent_node_id, bound_agent_session_ref)
  -> 返回会话详情
```

结果要求：
- 若团队未配置可用节点或底层会话绑定失败，则本次创建直接失败，不写入未绑定 Session。
- 会话创建成功后立即拥有明确的 `bound_agent_session_ref`。
- 新会话初始 `runtime_status = idle`。

### 10.2 会话空闲时发送消息

```text
成员发送消息
  -> MessageService 写入 member message(accepted)
  -> DispatchService 检查会话锁
  -> 无执行中任务：消息状态更新为 running
  -> Session.runtime_status 置为 running
  -> AdapterService 投递至 Codex
```

结果要求：
- 成员消息先持久化，再进入执行。
- 前端可立即看到该消息和会话运行中状态。

### 10.3 会话运行中继续发送消息

```text
成员发送消息
  -> MessageService 写入 member message(accepted)
  -> DispatchService 发现会话已 running
  -> 写入 Redis 队列
  -> Message.processing_status 置为 queued
  -> SessionEvent 写入 message.queued
  -> Session.runtime_status 保持 queued/running 聚合可见态
  -> WebSocket 推送排队中
```

结果要求：
- 新消息不抢占当前执行。
- 前端明确区分“当前正在运行的消息”和“排队中的消息”。

### 10.4 Codex 回传执行结果

```text
CodexAdapter 接收底层执行结果
  -> 归一化状态变化
  -> 写入 status.changed 事件
  -> 生成 command.summary 事件
  -> 写入 Agent 最终回复消息(is_final_reply=true)
  -> 更新当前消息为 completed
  -> 更新会话为 completed 或继续消费下一条排队消息
```

结果要求：
- 回放能看到状态变化和命令过程摘要。
- 搜索只命中最终回复，不命中命令过程摘要。

### 10.5 手动提取轻量 todo

```text
成员点击某条消息 -> 提取为 todo
  -> TodoService 校验来源消息可见性
  -> 写入 Todo(title, status=pending, source_message_id)
  -> 返回会话侧边栏 todo 列表
```

结果要求：
- todo 一定可跳回来源消息。
- todo 状态只允许待办/已完成两种。

### 10.6 搜索历史

```text
成员输入关键词
  -> SearchService 在 Message 索引中查询
  -> 仅返回 member message 与 agent final reply 命中
  -> 应用会话可见性过滤
  -> 返回命中内容与跳转定位信息
```

结果要求：
- 不返回事件层内容。
- 不返回当前成员无权查看的私有会话结果。

### 10.7 回放历史

```text
成员打开会话回放
  -> ReplayService 读取 Message + SessionEvent
  -> 按时间序组装消息、状态变化、命令摘要、可见性切换
  -> 应用可见性过滤
  -> 返回可读时间轴
```

结果要求：
- 回放范围严格限定为消息、状态变化、命令过程摘要、可见性切换。
- 不尝试还原底层终端的全量原始输出。

### 10.8 共享转私有

```text
成员触发共享转私有
  -> SessionService 校验操作者权限
  -> 更新 Session.visibility = private
  -> 写入 session.privatized 事件
  -> 写入 SessionAuditLog(共享起止时间、可见成员快照、操作人)
  -> 刷新会话可见索引
  -> 对其他成员隐藏会话
```

结果要求：
- 会话对其他成员立即不可见。
- 平台仍能证明该会话曾处于共享状态。

## 11. 逻辑接口清单

以下为概要设计层面的逻辑接口，后续详细 API 设计必须围绕这些输入输出展开，但不要求沿用相同命名。

| 接口 | 输入要点 | 输出要点 | 关键约束 |
|---|---|---|---|
| 创建团队 | 团队名称、创建者 | Team、成员关系 | 创建者默认管理员 |
| 创建项目 | `team_id`、项目名称 | Project | 项目属于单一团队 |
| 创建会话 | `project_id`、`visibility`、创建者 | Session、Codex 绑定引用 | 创建即绑定底层 Codex 会话 |
| 获取会话列表 | `project_id`、当前用户 | 会话列表、聚合状态 | 按可见性过滤 |
| 发送成员消息 | `session_id`、消息内容、发送者 | Message、当前排队/运行态 | 先落库，后调度 |
| 订阅会话实时流 | `session_id`、当前用户 | 消息、事件、状态变更推送 | 只推送当前用户可见内容 |
| 获取回放时间轴 | `session_id` | 消息+状态变化+命令摘要+可见性切换 | 不返回搜索外事件之外的原始过程 |
| 搜索会话历史 | 团队/项目范围、关键词、当前用户 | 仅消息层命中结果 | 只搜索成员消息和最终回复 |
| 提取 todo | `session_id`、`source_message_id`、标题 | Todo | 必须保留来源消息跳转 |
| 更新 todo 状态 | `todo_id`、状态 | Todo | 仅允许 pending/completed |
| 切换会话可见性 | `session_id`、目标可见性、操作者 | 更新后 Session | 共享转私有必须写回放事件与审计 |
| 获取会话审计记录 | `session_id` | 审计记录列表 | 不受当前可见性历史删除影响 |

## 12. Adapter 归一化映射

### 12.1 生命周期动作

MVP 阶段 Adapter 层建议统一提供以下动作语义：
- `startSession()`：启动新的底层 Codex 会话。
- `resumeSession()`：恢复已有底层会话引用。
- `sendMessage()`：向底层会话追加成员输入。
- `getStatus()`：查询当前执行状态。
- `dispose()`：释放绑定资源。
- `onEvent()`：向上层回传归一化后的执行事件。

`interrupt()` 可以保留为后续扩展接口，但 MVP 不应把中断抢占作为产品能力依赖。

### 12.2 底层信号到平台记录的映射

| 底层信号 | 平台记录 | 用途 |
|---|---|---|
| 会话创建/恢复成功 | `status.changed` | 会话初始化、恢复可见性 |
| 消息开始执行 | `status.changed` + message `running` | 前端展示运行中 |
| 新消息入队 | `message.queued` + message `queued` | 前端展示排队中 |
| 某段命令过程完成整理 | `command.summary` | 回放展示 |
| 最终回答产出 | agent message `is_final_reply=true` | 主消息流、搜索，不写 `message.final` 事件 |
| 执行失败 | `status.changed(error)` + 错误摘要 | 失败反馈与恢复入口 |
| 可见性变化 | `session.shared` / `session.privatized` | 回放时间轴和审计辅助 |

### 12.3 命令过程摘要约束

命令过程摘要必须面向人类阅读，用于说明本轮执行做了什么，而不是原样暴露底层终端所有输出。摘要至少要满足：
- 可解释本轮执行的主要动作。
- 能让协作者理解执行进度和结果脉络。
- 不参与搜索命中。
- 能在回放时间轴中独立展示。

## 13. 一致性与异常恢复约束

### 13.1 一致性规则

- 成员消息必须先写 PostgreSQL，再进入调度和 WebSocket 推送。
- 会话创建成功时必须同时具备 `bound_agent_node_id` 和 `bound_agent_session_ref`，绑定失败不得留下空壳会话。
- 会话执行锁必须按 `session_id` 维度隔离，防止并行污染上下文。
- 搜索索引只能从消息层构建，不能混入事件层。
- 回放时间轴必须从持久化后的消息层和事件层组装，不依赖运行时内存态。
- 共享转私有必须同时完成当前可见性更新、可回放事件写入和审计记录写入，三者缺一不可。

### 13.2 最小恢复规则

- Codex 执行失败时，会话状态更新为 `error`，并保留可读错误摘要。
- 已入队但尚未执行的消息不能因为当前消息失败而丢失。
- 单管理员节点不可用时，不做自动切换；新执行请求应返回节点不可用错误，并把相关会话标记为 `error` 或维持排队失败可见态。
- 会话恢复后允许继续消费既有队列，或由成员重新发送消息继续协作。
- 审计记录不因执行异常回滚丢失。

## 14. 可见性与权限控制

- 团队成员默认可访问团队内项目中的共享会话。
- 私有会话只对创建者可见，其他成员在会话列表、搜索、回放、todo 侧边栏中都不可见。
- 共享转私有后，平台保留回放时间线中的可见性切换节点和独立审计事实，但不再对其他成员暴露会话内容。
- 管理员负责执行节点维护；普通成员不承担节点配置职责。
- MVP 不做项目级复杂 ACL，但所有查询都必须统一套用会话可见性过滤。
- 当前用户必须来自鉴权上下文；在 JWT 完整落地前，可暂由测试 / 网关注入 `Authorization: Bearer <user_uuid>` 或 `x-syncai-user-id`，但实现层不得回退到 `creator_id` / `created_by` 充当当前用户。

## 15. 对后续测试与开发拆解的直接输入

### 15.1 测试设计应重点覆盖

- 会话创建与 Codex 绑定是否建立成功。
- 会话空闲执行、运行中入队、FIFO 消费是否正确。
- Agent 最终回复、状态变化、命令摘要是否按边界入库和展示。
- 搜索是否只命中成员消息和最终回复。
- 回放是否只展示消息、状态变化、命令过程摘要和可见性切换。
- todo 是否严格满足手动提取、两状态、来源跳转。
- 共享转私有后可见性变化、回放节点和审计记录是否同时成立。
- 单管理员节点异常时，错误可见性和排队消息保留是否成立。

### 15.2 开发拆解建议顺序

1. 先完成 Team / Project / Session / Message / Event / Todo / Audit 的基础数据模型。
2. 再实现会话调度、队列、执行锁、状态流转。
3. 然后接入 CodexAdapter，跑通一条真实的“成员消息 -> 最终回复”闭环。
4. 在闭环稳定后补齐回放、搜索、todo、共享转私有审计。
5. 最后补会话列表聚合态、实时推送和异常恢复细节。
