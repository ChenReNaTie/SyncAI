# SyncAI / 灵悉 AI - 产品需求文档

**版本：** v2.1  
**更新时间：** 2026-04-15  
**文档定位：** SyncAI 当前唯一产品基线文档。`概要设计`、`Agent接入设计`、测试设计与开发拆解均以本文为准。

---

## 1. 产品定义

SyncAI 不是重做一个 AI 编程工具，也不是重做 Codex 或 Claude Code。

SyncAI 的定位是：
**复用底层 Agent 的编程能力，做一个面向小团队的 AI 编程协作与共享平台。**

它解决的核心问题不是“个人如何更高效地用 AI 编程”，而是：

- 团队如何共享同一个 AI 会话历史
- 团队如何围绕同一个 Agent 持续协作
- 团队如何把过程沉淀为可搜索、可回放、可追踪的项目资产

一句话定义：
**把原本只存在于个人终端里的 AI 编程过程，转成团队可共享、可协作、可沉淀的项目资产。**

---

## 2. 第一性原则

当前产品定义必须同时满足以下原则：

- 价值中心是“共享会话 + 持续协作”，不是“替代底层 Agent”
- 第一阶段先做 Codex 的团队协作层，不做多 Agent 平台
- 所有成员围绕同一个底层会话继续追加输入，不复制多个并行副本
- 用户可见内容以“能读懂、能继续协作、能追溯”为标准，不以暴露所有底层原始输出为目标
- MVP 只做最短闭环，不把未拍板能力提前塞进第一版

---

## 3. 目标用户与核心问题

目标用户是 2 到 20 人的小团队，尤其是已经在使用 Codex，或后续计划引入 Claude Code 的开发团队。

当前核心问题：

- AI 会话历史只在个人本地，团队无法复用
- 同一项任务经常重复问、重复试、重复踩坑
- 团队成员无法看到 Agent 当前在做什么、做到哪一步
- 新成员接手项目时，缺少可追溯的 AI 协作上下文
- 讨论与执行过程分散在终端、聊天工具、手工 todo 中，无法形成一条主线

MVP 需要先跑通这条主链路：

**创建共享会话 -> 团队成员继续发消息 -> Agent 持续执行 -> 过程被同步展示 -> 结果可搜索 -> 消息可提取轻量 todo**

---

## 4. MVP 产品边界

### 4.1 组织结构

MVP 固定为三层结构：

- 团队
- 项目
- 会话

补充规则：

- 项目层保留，但只做轻量组织，不做复杂项目管理
- 一个团队可以有多个项目
- 一个项目可以有多个会话
- 第一版支持项目归档

### 4.2 团队角色

MVP 只区分两种角色：

- 管理员
- 普通成员

权限规则：

- 团队成员默认都能进入团队下的所有项目
- MVP 不做复杂的项目级权限隔离
- 只有管理员负责配置和维护当前团队所依赖的执行节点

### 4.3 会话类型

MVP 只有两种会话：

- 共享会话
- 私有会话

共享会话规则：

- 项目成员都可见
- 项目成员都可以继续发消息
- 新加入项目的成员，默认可看到该项目已有的共享会话历史

私有会话规则：

- 会话仍然挂在项目下
- 只有创建者本人可见

### 4.4 共享 / 私有切换

MVP 允许共享会话与私有会话互相转换。

其中共享转私有必须遵循以下规则：

- 转换后，对其他成员隐藏整个会话
- 平台保留审计记录
- 审计记录至少包含：共享开始时间、共享结束时间、当时可见成员范围
- 平台必须承认该会话历史上曾经被共享过，不能把它视为“从未共享过”

---

## 5. 会话协作规则

### 5.1 会话与底层 Agent 的关系

MVP 中，一个 SyncAI 会话一次只绑定一个底层 Agent 会话。

共享会话的本质不是复制多个副本，而是：
**团队成员围绕同一个底层 Agent 会话持续追加消息。**

### 5.2 第一阶段接入范围

第一阶段只接入 `Codex`。

`Claude Code` 只作为后续兼容目标保留，不纳入当前 MVP 必做范围。

### 5.3 单管理节点约束

MVP 采用单管理节点执行模式：

- 整个团队默认共享 1 个管理员本机上的 Codex 执行环境
- 平台负责接入、调度、展示和沉淀
- MVP 不做多管理员节点调度
- MVP 不做平台统一托管的多 Agent 执行集群

这是当前版本的明确产品约束，不是待讨论项。

### 5.4 运行中消息处理规则

当某个会话正在运行时：

- 后续成员消息不打断当前执行
- 新消息直接进入队列
- 前端必须明确展示“运行中”与“排队中”

MVP 不做复杂抢占、中断仲裁、多分支并行。

---

## 6. MVP 功能范围

### 6.1 会话核心能力

MVP 包含：

- 创建团队
- 创建项目
- 创建共享会话 / 私有会话
- 在会话中继续发送成员消息
- 展示 Agent 的最终回复
- 展示会话运行状态
- 展示排队中的后续消息
- 会话列表浏览
- 会话基础信息展示

MVP 暂不包含：

- IDE 插件
- 移动端 App
- 多 Agent 并行编排
- 多模型切换 UI
- 企业级复杂权限

### 6.2 Todo

MVP 只做轻量 todo，不做完整任务系统。

规则如下：

- todo 只能从消息中手动提取
- 不做 AI 自动建议 todo
- todo 只有两个字段：标题、状态
- 状态只有两个值：待办、已完成
- 每个 todo 都必须保留来源消息
- 点击 todo 可以跳回来源消息
- todo 展示在当前会话的侧边栏

### 6.3 搜索

MVP 搜索范围固定为：

- 成员消息
- Agent 最终回复

MVP 搜索明确不包含：

- 命令执行过程
- 状态事件
- 文件变化
- 其他过程事件

### 6.4 历史回放

MVP 回放范围固定为：

- 消息
- 状态变化
- 命令过程摘要

这里的“命令过程摘要”强调的是摘要展示，而不是把所有底层原始过程全部暴露出来。

### 6.5 展示原则

SyncAI 不要求把 Codex 终端样式 1:1 复刻到网页里。

但必须做到：

- Codex 的关键内容层级在 SyncAI 页面上都有对应表达
- SyncAI 可以使用自己的布局、视觉样式和信息层级

MVP 至少要展示：

- 成员消息
- Agent 最终回复
- 会话状态变化
- 命令过程摘要
- 排队状态

## 7. 典型场景

### 7.1 场景 A：多人围绕同一个会话持续推进

一个成员在项目下创建共享会话，并让 Codex 处理某个开发任务。执行过程中，其他成员进入同一会话查看上下文，并继续发送补充需求。

系统必须满足：

- 所有人围绕同一个底层 Codex 会话工作
- 正在运行时，后续成员消息进入队列
- 前端明确展示“运行中”和“排队中”
- 执行完成后，最终回复和过程摘要都能留在会话中继续追溯

### 7.2 场景 B：从消息中提取轻量 todo 跟进

会话里出现一个明确行动项，例如“补测试”“改接口”“确认部署变量”。成员不需要跳去另一个复杂任务系统，而是直接把该消息提取为 todo。

系统必须满足：

- todo 来自用户手动操作，而不是 AI 自动生成
- todo 仅包含标题和状态
- todo 侧边栏可见
- 点击 todo 能跳回原始消息，方便回到上下文继续协作

### 7.3 场景 C：回看历史并进行二次协作

一名新加入项目的成员，需要理解某个共享会话之前发生过什么，并在此基础上继续推进。

系统必须满足：

- 新成员可以看到历史共享会话
- 搜索只命中成员消息和 Agent 最终回复
- 回放中可以看到消息、状态变化和命令过程摘要
- 成员能够从命中位置或回放位置继续发消息，而不是重新开一个脱节的新上下文

### 7.4 场景 D：共享转私有后的可见性变化

某个共享会话在协作完成后，需要收回到创建者私有范围。

系统必须满足：

- 其他成员不再看到该会话
- 平台保留共享开始时间、共享结束时间和当时可见成员范围
- 该会话不能被系统当作“从未共享过”

---

## 8. 用户可见内容与平台内部记录

为避免产品口径和实现口径混淆，当前统一如下：

用户主要可见：

- 成员消息
- Agent 最终回复
- 会话状态变化
- 命令过程摘要
- 排队状态

平台内部可以记录更多结构化信息用于调度与审计，但这些内容不等于都要进入搜索或回放。

固定边界：

- 搜索只覆盖成员消息与 Agent 最终回复
- 回放只覆盖消息、状态变化与命令过程摘要

---

## 9. 核心用户流程

### 9.1 创建共享会话

1. 管理员创建团队并配置可用执行节点
2. 成员进入项目
3. 用户创建共享会话
4. 会话绑定底层 Codex 会话
5. 会话进入可协作状态

### 9.2 团队成员继续协作

1. 成员进入共享会话
2. 某个成员发送消息
3. 如果会话空闲，消息立即进入执行
4. 如果会话正在运行，新消息进入排队
5. 前端展示运行中 / 排队中状态
6. Agent 执行后返回结果
7. 平台展示最终回复并沉淀历史

### 9.3 提取 todo

1. 用户在某条消息上点击“提取为 todo”
2. 系统创建轻量 todo
3. todo 关联来源消息
4. todo 出现在当前会话侧边栏
5. 用户点击 todo 可跳回来源消息

### 9.4 搜索历史

1. 用户输入关键词
2. 系统只搜索成员消息与 Agent 最终回复
3. 返回结果包含所属项目、所属会话、命中内容、命中时间
4. 用户跳转到对应位置继续协作

### 9.5 共享转私有

1. 用户触发共享转私有
2. 系统修改会话可见性
3. 平台写入共享起止时间与可见成员范围审计
4. 其他成员不再看到该会话
5. 创建者仍可继续在私有状态下使用该会话

---

## 10. MVP 不做的内容

本阶段明确不做：

- 重新实现底层 AI 编程能力
- 多管理节点 / 多执行节点调度
- Claude Code 接入落地
- IDE 插件
- 自动化复杂任务流
- AI 自动提取 todo
- 企业级审批流
- 复杂项目权限模型
- 搜索命令过程 / 文件变化 / 其他事件
- 完整还原底层终端所有原始输出

---

## 11. 验收口径

如果以下条件成立，就说明当前产品定义可以作为开发落地基线：

- 团队可以在项目下创建共享会话和私有会话
- 项目成员可以围绕同一个共享会话继续发消息
- 会话运行时，后续消息进入队列，前端可见“排队中”
- 搜索只覆盖成员消息和 Agent 最终回复
- 历史回放只覆盖消息、状态变化和命令过程摘要
- todo 可以从消息手动提取，并跳回来源消息
- 共享转私有后，会话对其他成员隐藏，但审计记录仍保留
- 第一阶段基于单管理员本机上的 Codex 环境跑通完整闭环

---

## 12. 当前结论

SyncAI 当前 MVP 的本质不是“再做一个 AI IDE”，而是：

**在 Codex 之上，先做一个适合小团队共享与协作的会话层、调度层和沉淀层。**

只要这条主链路稳定跑通，后续再接 Claude Code、扩权限、扩节点、扩插件，才有意义。

---

## 13. 背景判断

以下结论已吸收早期竞品调研与商业化方案文档，只保留当前 MVP 真正有用的判断。

### 13.1 竞品空白

- 现有 AI 编程工具大多优化个人效率，AI 会话历史通常停留在个人本地。
- 通用协作工具虽然支持聊天或 AI 插件，但不围绕同一个 Agent 会话持续协作。
- 团队版 AI 工具更多是在共享账号或工作区，仍缺少“共享执行过程、共享上下文、共享后续跟进”的闭环。

### 13.2 对 MVP 的直接约束

- 第一阶段优先服务 2 到 20 人的小团队，先把共享会话、协作续写、搜索回放、轻量 todo 跑通。
- 商业化更适合按团队或席位订阅，而不是按单次对话计费，因此 MVP 更应优先沉淀协作价值，而不是过早设计复杂计费体系。
- 企业级权限、分析报表、私有化部署可以作为后续扩展，但不应反向挤占当前 MVP 范围。
