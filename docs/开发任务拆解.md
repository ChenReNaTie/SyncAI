# SyncAI / 灵悉 AI - 开发任务拆解

**版本：** v1.1  
**更新时间：** 2026-04-16  
**文档定位：** 基于 `docs/概要设计.md`、`docs/数据库设计.md`、`docs/API设计.md`、`docs/测试用例设计.md` 输出的正式开发拆解文档，用于指导正式开编码前后的实施顺序、里程碑和交付标准。

---

## 1. 拆解原则

当前拆解不再沿用旧版 `07-mvp-development-plan.md` 的工作区、任务卡片、知识库、多 Agent 范围，而是严格围绕已拍板 MVP：

- 团队 > 项目 > 会话
- 第一阶段只支持 Codex
- Agent 默认运行在管理员本机
- todo 只做轻量版
- 搜索第一版只搜消息
- 会话创建成功即绑定团队当前 Codex 节点
- 回放时间轴包含可见性切换节点

任务拆解遵循四条原则：

- 先把“能跑通最小闭环”的主链路做出来，再补阅读与沉淀能力。
- 先稳定后扩展，优先落数据模型、调度和 Adapter，不先做花哨界面。
- 先用 Mock Adapter 保证开发速度，再接真实 Codex 环境联调。
- 所有任务都要能对照已有文档和测试用例落验收。

---

## 2. 当前起点

当前仓库状态：

- `docs/` 已完成需求、架构、接入、概要、测试基线整理。
- 仓库还没有正式 `src/`、`server/`、`web/` 或 `tests/` 代码骨架。
- 还未开始数据库迁移、API 实现、前端界面和真实 Codex 联调。

因此，后续任务不是“继续讨论需求”，而是从工程脚手架和最小闭环直接开工。

---

## 3. 里程碑

| 里程碑 | 目标 | 完成标志 |
|---|---|---|
| M0 | 工程骨架可运行 | 本地能启动后端、前端、PostgreSQL、Redis |
| M1 | 核心数据与基础接口可用 | 团队、项目、会话、消息表和基础 CRUD 打通 |
| M2 | 会话执行闭环跑通 | 成员消息 -> 调度 -> CodexAdapter -> 最终回复 |
| M3 | 阅读与沉淀能力补齐 | 回放、搜索、todo、共享转私有审计上线 |
| M4 | 联调与验收完成 | 关键测试用例跑通，可进入正式开发迭代 |

## 4. 分阶段任务

### 4.1 Phase 0：开工准备与工程骨架

目标：

- 建立可持续开发的工程基础

| ID | 任务 | 产出 | 前置 |
|---|---|---|---|
| DEV-001 | 初始化仓库代码结构 | `server/` `web/` `packages/` 或等价目录 | 无 |
| DEV-002 | 配置本地开发环境 | `docker-compose.yml`、`.env.example`、启动脚本 | DEV-001 |
| DEV-003 | 建立后端基础框架 | Fastify 或等价服务骨架、日志、配置加载 | DEV-001 |
| DEV-004 | 建立前端基础框架 | React + Vite 或等价骨架、路由、状态管理骨架 | DEV-001 |
| DEV-005 | 建立基础质量工具链 | ESLint、Prettier、提交规范、基础 CI | DEV-001 |

验收标准：

- 本地可一键启动 PostgreSQL、Redis。
- 后端健康检查可访问。
- 前端基础页面可访问。
- 环境变量模板可供团队共享。

### 4.2 Phase 1：数据库与基础业务骨架

目标：

- 把 `docs/数据库设计.md` 落成迁移和基本数据访问层

| ID | 任务 | 产出 | 前置 |
|---|---|---|---|
| DEV-101 | 实现数据库迁移框架 | ORM 或 SQL migration 基础设施 | Phase 0 |
| DEV-102 | 建立核心表 | `users` `teams` `team_members` `team_agent_nodes` `projects` `sessions` `messages` `session_events` `todos` `session_audit_logs` | DEV-101 |
| DEV-103 | 建立状态枚举与约束 | CHECK / ENUM / 索引 | DEV-102 |
| DEV-104 | 实现基础仓储层 | Team / Project / Session / Message / Todo / Audit Repository | DEV-102 |
| DEV-105 | 实现认证与当前用户上下文 | 登录、JWT、中间件 | DEV-101 |
| DEV-106 | 实现团队与项目基础接口 | 团队 CRUD、成员管理、项目 CRUD | DEV-104, DEV-105 |
| DEV-107 | 实现管理员节点配置与会话创建基础接口 | `GET/PUT /teams/{id}/agent-node`、会话创建时即时绑定当前节点 | DEV-104, DEV-105 |
| DEV-108 | 实现会话详情与列表接口 | 会话详情、列表、可见性过滤 | DEV-104, DEV-105, DEV-107 |

验收标准：

- 数据迁移可从零建库。
- 能配置团队当前 Codex 节点。
- 能创建团队、项目、共享会话、私有会话，且创建成功即返回 `bound_agent_session_ref`。
- 无可用节点或绑定失败时，会话创建直接失败，不生成空壳会话。
- 会话列表正确体现可见性过滤。

### 4.3 Phase 2：消息、调度与 Mock Adapter 闭环

目标：

- 先在不依赖真实 Codex 的前提下跑通执行闭环

| ID | 任务 | 产出 | 前置 |
|---|---|---|---|
| DEV-201 | 实现消息发送接口 | `POST /sessions/{id}/messages` | Phase 1 |
| DEV-202 | 建立会话执行锁与队列 | Redis 锁、队列管理、恢复扫描 | DEV-201 |
| DEV-203 | 实现消息状态流转 | `accepted/queued/running/completed/failed` | DEV-202 |
| DEV-204 | 实现会话运行状态聚合 | `idle/queued/running/completed/error` | DEV-202 |
| DEV-205 | 实现 Mock CodexAdapter | 支持 `startSession`、可控成功、长执行、失败、节点不可用 | DEV-202 |
| DEV-206 | 打通 Mock 闭环 | 会话创建即绑定 -> 成员消息 -> 排队/执行 -> 事件 -> 最终回复 | DEV-203, DEV-205 |
| DEV-207 | 建立基础 WebSocket 推送 | 会话订阅、消息推送、状态推送 | DEV-206 |

验收标准：

- 会话创建成功即拥有 Mock `bound_agent_session_ref`。
- 空闲会话消息能立即进入执行。
- 运行中新增消息能进入队列并显示排队中。
- Mock Adapter 能生成最终回复和命令摘要。

### 4.4 Phase 3：真实 Codex 接入

目标：

- 把 Mock 能力替换为真实管理员本机 Codex 接入

| ID | 任务 | 产出 | 前置 |
|---|---|---|---|
| DEV-301 | 实现真实管理员节点接入注册 | 真实节点注册、配置复用、连接校验 | Phase 2 |
| DEV-302 | 实现节点心跳与状态上报 | `offline/connecting/online/error` | DEV-301 |
| DEV-303 | 实现真实 `CodexAdapter` 生命周期 | `startSession` `resumeSession` `sendMessage` `getStatus` `dispose` | DEV-206 |
| DEV-304 | 建立 Adapter 事件归一化 | `status.changed` `command.summary` `final.reply` `message.failed` | DEV-303 |
| DEV-305 | 跑通真实执行闭环 | 真实 Codex 最终回复入库和推送 | DEV-302, DEV-304 |
| DEV-306 | 节点异常与恢复处理 | 节点不可用错误、恢复后继续协作 | DEV-305 |

验收标准：

- 一个团队只绑定一个管理员本机 Codex 节点。
- 真实环境下会话创建成功即返回真实 `bound_agent_session_ref`。
- 真实环境下可跑通“发送消息 -> 收到最终回复”。
- 节点异常时用户能看到明确错误，不是假成功。

### 4.5 Phase 4：回放、搜索、todo、审计

目标：

- 把“能执行”扩展为“能阅读、能回看、能沉淀”

| ID | 任务 | 产出 | 前置 |
|---|---|---|---|
| DEV-401 | 实现回放接口 | `GET /sessions/{id}/replay` | Phase 2 |
| DEV-402 | 实现命令摘要持久化 | `session_events(command.summary)` | DEV-206 |
| DEV-403 | 实现消息搜索 | `GET /teams/{id}/search`，只搜消息层 | Phase 1 |
| DEV-404 | 实现 todo 接口 | 创建 todo、更新状态、侧栏查询 | Phase 1 |
| DEV-405 | 实现共享 / 私有切换与审计 | 可见性更新、审计记录、回放事件写入 | Phase 1 |
| DEV-406 | 实现共享转私有后的统一过滤 | 列表、详情、搜索、回放、todo、WS 一致失效 | DEV-405 |

验收标准：

- 回放只展示消息、状态变化、命令过程摘要、可见性切换。
- 搜索只命中成员消息和 Agent 最终回复。
- todo 只能手动提取，且能跳回来源消息。
- 共享转私有后内容不可见，但审计与回放节点仍可追溯。

## 5. 推荐实施顺序

如果只有 1 名开发者，建议严格按以下顺序推进：

1. Phase 0
2. Phase 1
3. Phase 2
4. Phase 3
5. Phase 4
6. Phase 5
7. Phase 6

如果有 2 名开发者，可在以下节点并行：

- 一人负责数据库、认证、团队 / 项目 / 会话基础接口
- 一人负责 WebSocket、调度器、Mock Adapter

但以下工作不建议并行抢跑：

- 未有数据模型前先写搜索
- 未有 Mock 闭环前先接真实 Codex
- 未有后端稳定接口前先深做前端细节

## 7. 完成定义

进入“可以正式扩展开发”的标准不是页面做完，而是以下条件同时成立：

- 团队、项目、共享会话、私有会话都能创建与查询。
- 会话消息能按串行规则执行，运行中消息能排队。
- Mock 和 Real CodexAdapter 都能接入。
- 会话创建成功即绑定当前 Codex 节点，不存在延后绑定。
- 最终回复、状态变化、命令摘要、可见性切换节点能被正确写入并展示。
- 搜索、回放、todo、共享转私有审计符合既定边界。
- 对照 `docs/测试用例设计.md` 的 P0 用例通过。
