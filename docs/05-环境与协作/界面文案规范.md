# 界面文案规范

## 目的

统一 SyncAI 前端界面的文案口径，避免中英文混用、术语漂移和同义词并存，保证界面一致性、可理解性和可维护性。

## 总原则

1. 面向普通用户的界面文案，默认使用中文。
2. 只有在英文更准确、更短、或属于行业约定时，才保留英文。
3. 同一个概念在全站只保留一个主称呼，不要同页出现多个近义写法。
4. 用户看到的是“产品界面”，不是“内部实现”，不要把后端字段名、状态码、模型事件名直接暴露出来。
5. 快捷键、代码、路径、协议名、产品名按原样保留，不强行翻译。

## 必须使用中文的场景

以下内容默认必须中文：

- 页面标题：如“团队”“项目”“会话列表”“消息区”
- 导航与返回：如“返回项目”“返回团队”“返回仪表盘”
- 按钮文案：如“发送消息”“查看回放”“创建项目”“标记完成”
- 表单标签与占位文案：如“项目名称”“输入关键词搜索...”
- 空状态、错误提示、成功提示、帮助说明
- 用户可见状态：如“处理中”“已完成”“失败”“空闲”“在线”“离线”
- 统计卡片标题：如“消息总量”“待办进度”“最后更新”
- 列表分区标题：如“实时处理流”“协作待办”“团队成员”

## 可以保留英文的场景

以下内容允许保留英文，并建议保持原样：

- 快捷键与按键名：`Enter`、`Shift + Enter`、`Esc`、`Tab`
- 品牌名或产品名：`SyncAI`、`Codex`
- 技术缩写：`API`、`WebSocket`、`UUID`、`URL`
- 代码实体：变量名、接口名、事件名、状态码、路由路径
- 文件路径与目录名：如 `web/src/pages/SessionPage.tsx`
- 用户明确熟悉且行业通用的短词，但前提是不影响理解

## 中英混排规则

适合使用“英文术语 + 中文说明”的场景：

- 快捷键提示：`Enter 发送 / Shift + Enter 换行`
- 品牌或技术名首现说明：`Codex 会话线程`
- 无法避免的专有名词：`WebSocket 实时连接`

不建议使用以下混搭方式：

- `Conversation 消息区`
- `Action Items 待办区`
- `Live Trace 实时处理流`
- `Session Workspace 会话工作台`

原因是这类写法会增加视觉噪音，但不增加理解收益。

## 术语主词典

### 页面与导航

- `Dashboard` -> `仪表盘`
- `Team` -> `团队`
- `Project` -> `项目`
- `Session` -> `会话`
- `Back` -> `返回`

### 会话协作

- `Conversation` -> `会话消息`
- `Message` -> `消息`
- `Input` -> `输入区`
- `Replay` -> `回放`
- `Action Items` -> `协作待办`
- `Todo` -> `待办`
- `Live Trace` -> `实时处理流`
- `Thread` -> `线程`
- `Agent reply` -> `助手回复`
- `Agent node` -> `助手节点`

### 状态类

- `queued` -> `排队中`
- `pending` -> `等待中`
- `running` / `in_progress` -> `处理中`
- `completed` -> `已完成`
- `failed` -> `失败`
- `idle` -> `空闲`
- `online` -> `在线`
- `offline` -> `离线`
- `shared` -> `共享`
- `private` -> `私有`

## 文案风格规则

### 标题

- 简短直接，优先 2 到 6 个汉字
- 避免“管理中心”“综合工作台”这类空泛大词
- 示例：`会话列表`、`团队成员`、`待办区`

### 按钮

- 使用动词开头
- 一个按钮只表达一个动作
- 示例：`发送消息`、`创建项目`、`退出回放`

### 提示文案

- 先说结果，再说原因或下一步
- 示例：`还没有消息，先发出第一条协作指令。`
- 避免：`No data found`、`Operation failed`

### 错误提示

- 优先给出用户可理解描述，不直接展示内部错误码
- 示例：`加载会话失败`、`发送消息失败`
- 如果需要补充细节，放在后半句，不要只有技术词

### 状态文案

- 尽量使用“中性、可感知、可扫描”的短词
- 优先两到四字：`处理中`、`已完成`、`失败`
- 不使用生硬直译：如 `进行中`、`执行完成状态`

## 禁止项

以下写法应避免：

- 同一概念多种叫法并存，如“Agent / AI / 助手”混用
- 直接暴露内部枚举值，如 `completed`、`queued`
- 一半中文一半英文但没有必要的标题
- 过长按钮文案，如“点击这里开始创建一个新的项目”
- 生硬机翻式文案，如“实时追踪流区域”

## 当前项目的推荐口径

当前项目建议统一采用以下口径：

- 对普通用户：优先中文
- 对快捷键：保留英文按键名
- 对产品名：保留 `SyncAI`、`Codex`
- 对协作角色：统一使用“成员”“助手”，不要混用“AI”“Agent”
- 对系统状态：界面展示中文，内部代码保留英文枚举

## 执行规则

后续修改前端页面时，按以下顺序判断：

1. 这是用户直接看到的内容吗？如果是，优先中文。
2. 这是快捷键、品牌名、路径、代码名吗？如果是，保留英文。
3. 这是一个已经定义过的术语吗？如果是，必须复用词典中的固定写法。
4. 如果拿不准，宁可更短、更直白，不要写成中英混合标题。

## 变更建议

后续如果要批量清理界面文案，建议按下面顺序执行：

1. 先改页面标题、按钮、返回文案
2. 再改状态标签和空状态文案
3. 最后清理说明文字、提示语和异常提示

这样可以先统一用户最常见、最显眼的文案，再处理细节。
