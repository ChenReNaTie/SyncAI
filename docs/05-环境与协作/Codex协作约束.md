# Codex协作约束

## 适用范围

本文件用于 `SyncAI / 灵悉 AI` 项目的 `Codex` 长期主会话启动、续跑、追加任务与阶段切换。

目标不是只让 `Codex` "知道项目在做什么"，而是确保它在整个执行过程中遵守统一的评审、测试和交付规则。

## 总原则

1. 这是持续项目。
2. 长任务执行过程中，除非出现明显偏差、危险操作或实质性阻塞，否则不要轻易中断。
3. 不允许为了追求更快汇报而跳过验证、压缩评审或提前把半成品当完成品。
4. 一切偏差、取舍、争议与实现判断，统一以需求文档为准。

## 分阶段门禁

### 阶段 1：概要设计

- 概要设计产出必须完成 3 次评审。
- 评审目标包括但不限于：系统分层、模块边界、接口边界、数据模型、状态流、部署形态、测试策略入口。
- 评审中发现的问题，如果涉及产品取舍、边界裁剪、规则冲突或口径不一致，必须汇总为待拍板项，由阿铁转交拿铁决策。
- 未完成 3 次评审前，不得把概要设计视为定稿，也不得直接跳到“开发完成”的叙述。

### 阶段 2：测试设计

- 必须先基于概要设计产出测试用例，再基于测试用例产出测试脚本或测试脚本设计。
- 测试用例必须完成 2 次审查。
- 测试脚本或测试脚本设计必须完成 2 次审查。
- 审查重点包括：覆盖范围、口径一致性、可执行性、是否直接对应需求文档与概要设计。

### 阶段 3：TDD 开发与验证

- 正式开发必须按测试驱动开发推进。
- 代码完成的判定标准不是“代码写完”，而是“相关测试通过”。
- 测试发现的问题不能只做表面修补，必须追查根因。
- 若实现结果与需求文档、概要设计、测试设计存在冲突，应先回到文档基线核对，不得自行漂移需求口径。

## 汇报要求

每次阶段性汇报时，至少明确以下内容：

1. 当前处于哪个阶段：`概要设计评审` / `测试设计与审查` / `TDD 开发与验证`
2. 本轮新增或修改了哪些真实文件
3. 已完成了第几轮评审或审查
4. 是否存在必须由拿铁拍板的阻塞项
5. 若无阻塞，下一步准备推进什么

## 禁止事项

- 不要把“已有初稿”汇报成“已经完成”。
- 不要把“测试未过的代码”汇报成“开发完成”。
- 不要重复讨论已经拍板的产品结论。
- 不要在没有依据的情况下新增产品能力或修改既定边界。

## 当前项目基线提醒

当前 `SyncAI` 项目已确认的开发基线，仍以现有正式中文文档为准，尤其包括：

- `docs/01-产品/产品需求文档.md`
- `docs/02-架构设计/Agent接入设计.md`
- `docs/02-架构设计/概要设计.md`（若后续更新，以最新评审通过版本为准）

在任何实现或评审中，如与这些文档冲突，应优先标记问题并回到文档基线核对。
