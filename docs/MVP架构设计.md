# SyncAI / 灵悉 AI - MVP 架构设计

**版本：** v2.1  
**更新时间：** 2026-04-15  
**文档定位：** 基于《产品需求文档》的 MVP 技术基线，明确系统边界、模块划分、关键数据流、状态流与落地约束。

---

## 1. 架构目标

本架构只服务当前 MVP，不追求一步到位的大而全平台。

当前只解决 5 件事：

- 团队围绕同一个 Agent 会话协作
- 会话运行过程可以实时同步
- 结果可以沉淀为可搜索、可回放的历史
- 消息可以手动提取轻量 todo
- 共享 / 私有切换具备基础审计能力

对应的架构原则：

- 先跑通主闭环，不为未来能力过度设计
- 平台内部统一事件模型，但用户可见范围严格受产品边界约束
- 同一个会话按串行调度执行，避免底层上下文混乱
- MVP 优先单体应用，不拆微服务
- 明确依赖单管理员本机上的 Codex 执行环境

---

## 2. 总体架构

MVP 推荐采用“单体应用 + 实时通信 + 本地 Codex 节点”的架构。

```text
Web 前端
  -> HTTP / WebSocket
应用服务
  -> PostgreSQL
  -> Redis
  -> CodexAdapter
管理员本机 Codex 执行环境
```

各层职责如下：

- Web 前端：展示团队、项目、会话、消息、状态、命令摘要、todo、搜索结果
- 应用服务：承载业务逻辑、调度逻辑、事件落库、搜索查询、回放组装
- PostgreSQL：存储业务主数据与会话消息 / 事件 / 审计
- Redis：承担会话队列、执行锁、实时态缓存
- 管理员本机 Codex 环境：实际执行底层 Agent 会话

---

## 3. 关键架构约束

### 3.1 单管理节点

当前 MVP 默认整支团队共享 1 个管理员本机上的 Codex 执行环境。

这意味着：

- 平台只需要对接一个活跃执行节点
- 暂不处理多节点路由
- 暂不处理节点间会话迁移
- 暂不处理跨管理员的执行一致性问题

### 3.2 一个 SyncAI 会话绑定一个底层 Agent 会话

当前一个 SyncAI 会话一次只绑定一个底层 Agent 会话。

共享会话的继续协作，采用对同一底层会话追加输入的方式，不生成多个副本。

### 3.3 串行调度

虽然多个成员可以向同一个共享会话连续发消息，但底层执行必须串行。

因此架构必须保证：

- 同一会话同一时间只执行一个任务
- 运行中的会话继续收到消息时，新消息进入队列
- 前端能感知该消息处于“排队中”

### 3.4 搜索与回放范围固定

MVP 内部可以存储必要的状态与命令摘要，但产品可见口径固定如下：

- 搜索只覆盖成员消息与 Agent 最终回复
- 回放只覆盖消息、状态变化、命令过程摘要

架构设计不能偏离这个口径。

## 4. 模块划分

### 4.1 前端模块

前端建议采用三栏布局：

- 左侧：团队 / 项目 / 会话导航
- 中间：消息流、状态变化、命令过程摘要
- 右侧：todo 侧栏、会话信息

前端职责：

- 提交成员消息
- 展示会话运行状态与排队状态
- 展示 Agent 最终回复
- 展示回放内容
- 提供 todo 提取入口
- 提供搜索入口与结果跳转

### 4.2 会话服务

会话服务负责：

- 创建与查询会话
- 维护共享 / 私有属性
- 处理共享 / 私有转换
- 记录转换审计信息
- 提供会话列表数据

### 4.3 消息服务

消息服务负责：

- 写入成员消息
- 写入 Agent 最终回复
- 关联消息与会话
- 为搜索提供消息层索引来源

### 4.4 事件服务

事件服务负责：

- 记录会话状态变化
- 记录命令过程摘要
- 记录排队状态等实时过程事件
- 为回放组装提供结构化素材

注意：

- MVP 标准回放只消费消息、状态变化、命令过程摘要
- 搜索不直接消费事件层内容

### 4.5 Todo 服务

Todo 服务负责：

- 从指定消息手动创建 todo
- 管理 todo 状态
- 维护 todo 与来源消息的关联
- 返回侧栏展示数据

### 4.6 搜索服务

搜索服务负责：

- 建立成员消息与 Agent 最终回复的搜索索引
- 提供项目级 / 全局搜索
- 返回命中消息及其会话、项目上下文

MVP 不搜索命令、状态事件、文件变化。

### 4.7 调度服务

调度服务是 MVP 的关键模块，负责：

- 维护每个会话的执行状态
- 把消息送入会话队列
- 按串行规则消费队列
- 驱动底层 `CodexAdapter`
- 把执行过程转成平台内部统一记录

### 4.8 Agent 适配层

MVP 第一阶段只实现：

- `CodexAdapter`

它负责：

- 启动或恢复底层 Codex 会话
- 发送成员输入
- 捕获状态与输出
- 转换为平台可用的数据结构

## 5. 关键数据流

### 5.1 会话创建流

```text
前端创建会话
  -> 会话服务写入 Session
  -> 调度服务初始化会话运行态
  -> CodexAdapter 建立底层会话绑定
  -> 返回会话详情
```

### 5.2 消息执行流

```text
成员发送消息
  -> 消息服务写入 member message
  -> 调度服务检查 session lock
  -> 空闲: 直接投递 CodexAdapter
  -> 运行中: 写入队列 + 记录 queued 事件
  -> WebSocket 推送运行态 / 排队态
```

### 5.3 Agent 结果回传流

```text
CodexAdapter 接收底层输出
  -> 调度服务归一化
  -> 事件服务写入状态变化 / 命令摘要
  -> 消息服务写入最终回复
  -> WebSocket 推送给在线成员
  -> 搜索服务消费消息层索引
```

### 5.4 todo 提取流

```text
用户点击提取 todo
  -> Todo 服务读取 source_message_id
  -> 写入 Todo
  -> 返回侧栏数据
  -> 前端支持跳回来源消息
```

### 5.5 共享转私有流

```text
用户触发共享转私有
  -> 会话服务更新可见性
  -> 审计日志写入共享起止时间与可见成员快照
  -> 会话索引刷新
  -> 其他成员前端不再看到该会话
```

## 6. 关键状态流

### 6.1 会话运行状态

MVP 建议统一会话运行状态：

- `idle`
- `queued`
- `running`
- `completed`
- `error`

说明：

- `queued` 表示当前会话已有待执行成员消息
- `running` 表示底层 Codex 正在执行
- `completed` 表示最近一次执行结束，可回落为 `idle`
- `error` 表示最近一次执行失败，需要保留错误摘要与可恢复入口

### 6.2 单条消息处理状态

成员消息建议具备独立处理态：

- `accepted`
- `queued`
- `running`
- `completed`
- `failed`

原因：

- 会话级状态适合做全局展示
- 消息级状态适合做“这条消息现在排到哪一步了”的精确反馈

### 6.3 共享可见性状态

会话可见性只允许：

- `shared`
- `private`

其中共享转私有不是删除历史，而是：

- 变更当前可见性
- 新增一条完整审计记录

## 7. 数据模型

### 7.1 核心实体

MVP 至少需要以下核心实体：

- `User`
- `Team`
- `TeamMember`
- `Project`
- `Session`
- `Message`
- `SessionEvent`
- `Todo`
- `SessionAuditLog`

### 7.2 关键字段建议

`Session` 至少包含：

- `id`
- `project_id`
- `type`，取值为 `shared` / `private`
- `creator_id`
- `bound_agent_type`
- `bound_agent_session_ref`
- `runtime_status`
- `archived_at`

`Message` 至少包含：

- `id`
- `session_id`
- `sender_type`，取值为 `member` / `agent`
- `sender_id` 或 agent 标识
- `content`
- `processing_status`
- `is_final_reply`
- `created_at`

`SessionEvent` 至少包含：

- `id`
- `session_id`
- `event_type`
- `summary`
- `payload`
- `created_at`

`Todo` 至少包含：

- `id`
- `session_id`
- `source_message_id`
- `title`
- `status`
- `creator_id`

`SessionAuditLog` 至少包含：

- `id`
- `session_id`
- `action_type`
- `visible_scope_snapshot`
- `started_at`
- `ended_at`
- `operator_id`

其中：

- 搜索主要读取 `Message`
- 回放读取 `Message + SessionEvent`
- 共享转私有审计读取 `SessionAuditLog`

## 8. 统一记录模型

虽然前端展示的是“消息流”，但系统内部需要严格区分两类内容：

- 消息
- 事件

### 8.1 消息

消息是用户能直接阅读和继续协作的主体内容，MVP 只包含：

- 成员消息
- Agent 最终回复

### 8.2 事件

事件是系统结构化记录，MVP 回放主要使用：

- 状态变化事件
- 命令过程摘要事件
- 排队状态事件
- 可见性切换事件

建议统一事件类型至少包括：

- `status.changed`
- `command.summary`
- `message.queued`
- `session.shared`
- `session.privatized`

注意：

- `message.queued` 主要服务实时状态与排队展示
- 搜索不使用事件
- 回放不要求展示所有原始过程，只展示产品定义的摘要层

---

## 9. 模块边界与接口边界

为了避免前后端和 Adapter 层耦合失控，当前边界必须固定：

- 前端只消费平台定义的消息、事件、todo、搜索结果结构
- 业务层不直接依赖 Codex 原始终端输出格式
- 只有 `CodexAdapter` 可以与底层 Codex 运行时直接交互
- 搜索服务只读取消息层，不跨到事件层做模糊扩张
- 回放服务只读取产品定义允许暴露的记录类型

推荐内部接口分层：

- `SessionService`: 会话创建、查询、可见性切换
- `MessageService`: 消息写入、消息状态更新
- `DispatchService`: 队列、执行锁、任务投递
- `AdapterService`: 管理底层 Agent 生命周期
- `ReplayService`: 按产品口径组装回放流
- `SearchService`: 只索引成员消息与 Agent 最终回复

---

## 10. 部署形态

MVP 推荐部署形态：

- 应用服务部署在一台中心服务器
- PostgreSQL 与 Redis 可与应用服务同机或近邻部署
- 管理员本机安装并维护 Codex 运行环境
- 平台通过受控接入层与管理员本机上的 Codex 交互

第一版不引入：

- 多节点执行集群
- 统一托管的云端 Agent 池
- 跨管理员自动故障切换

---

## 11. 测试入口

当前架构已经足够支撑后续测试设计，重点测试入口应围绕以下主链路展开：

- 共享会话创建与继续协作
- 运行中消息入队与“排队中”展示
- Agent 最终回复与命令摘要写入
- 搜索仅命中成员消息与 Agent 最终回复
- 回放仅展示消息、状态变化、命令过程摘要
- todo 手动提取与来源消息跳转
- 共享转私有后的可见性变化与审计留痕
- 单管理节点下的串行调度与恢复

---

## 12. 当前架构是否可落地

以当前范围来看，这套架构已经可以作为开发基线。

原因是它已经明确了：

- 产品边界
- 执行边界
- 单节点约束
- 队列规则
- 搜索范围
- 回放范围
- 共享转私有的审计要求
- 前后端与 Adapter 的职责边界

下一步应该进入接口定义、数据表设计、测试设计和开发任务拆解，而不是继续反复改产品口径。
