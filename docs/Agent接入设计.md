# SyncAI / 灵悉 AI - Agent 接入设计

**版本：** v2.1  
**更新时间：** 2026-04-15  
**文档定位：** 说明 SyncAI 在 MVP 阶段如何接入 Codex，如何把底层执行过程转成团队可协作、可展示、可回放的会话能力。

---

## 1. 设计目标

本设计只解决一个问题：

**如何把 Codex 接进 SyncAI，并让团队成员围绕同一个底层会话持续协作。**

当前文档不解决：

- 多 Agent 并行编排
- 多执行节点路由
- Claude Code 的落地实现
- 终端像素级镜像复刻

---

## 2. MVP 接入边界

### 2.1 当前只接入 Codex

MVP 第一阶段只实现 `CodexAdapter`。

`Claude Code` 只保留后续扩展接口，不进入本阶段开发范围。

### 2.2 会话绑定规则

一个 SyncAI 会话一次只绑定一个底层 Agent 会话。

共享会话中的多成员协作，实质上是对同一个底层 Codex 会话持续追加输入，而不是派生多个并发副本。

### 2.3 单管理节点规则

MVP 默认使用单管理节点模式：

- 整个团队先共享 1 个管理员本机上的 Codex 执行环境
- SyncAI 平台只负责接入、调度、展示和存档
- MVP 不处理多管理员、多节点、多环境切换

这是硬约束，不是优化项。

### 2.4 运行中消息处理规则

当某个会话已经在运行时：

- 新成员消息直接进入队列
- 不抢占当前执行
- 前端明确展示“排队中”

---

## 3. 接入原则

### 3.1 内容结构一致，不做终端镜像

SyncAI 需要对齐 Codex 的关键内容层级，但不要求终端界面 1:1 复刻。

MVP 要求对齐的层级包括：

- 成员消息
- Agent 最终回复
- 会话状态变化
- 命令过程摘要
- 排队状态

### 3.2 平台内部统一接入层

前端和业务层不能直接依赖 Codex 的原始输出格式。

正确做法是：

- 由 `CodexAdapter` 负责接入底层会话
- 由接入层把底层输出整理成平台记录
- 前端只消费平台定义的数据结构

### 3.3 调度优先于花哨展示

SyncAI 的核心不是“显示得像不像终端”，而是“多人继续发消息时，系统是否还能稳定串行执行并保留上下文”。

因此：

- 调度正确性优先
- 记录一致性优先
- 展示细节服从产品口径

---

## 4. 运行拓扑

```text
成员消息
  -> SyncAI 会话服务
  -> 调度服务
  -> CodexAdapter
  -> 管理员本机 Codex

Codex 输出
  -> CodexAdapter
  -> 状态 / 摘要 / 最终回复
  -> 数据库存储
  -> WebSocket 推送
  -> 搜索 / 回放 / todo 等上层能力消费
```

这条链路里，`CodexAdapter` 是边界层，负责隔离 SyncAI 与底层运行细节。

---

## 5. CodexAdapter 设计

### 5.1 责任范围

`CodexAdapter` 负责：

- 启动或恢复底层 Codex 会话
- 接收来自调度服务的成员输入
- 监听底层会话执行状态
- 采集最终回复
- 生成命令过程摘要
- 向上层回传统一结构的数据

### 5.2 生命周期语义

平台内部建议统一以下生命周期语义：

- `startSession()`
- `resumeSession()`
- `sendMessage()`
- `getStatus()`
- `interrupt()`，MVP 可预留但不强依赖
- `dispose()`
- `onEvent()`

重点不是接口命名本身，而是生命周期要统一，便于后续兼容其他 Agent。

### 5.3 输出捕获优先级

推荐优先级如下：

1. 结构化输出或可解析事件流
2. 可解析的会话文件 / 日志
3. CLI 输出解析

MVP 不建议把“纯终端镜像”作为主数据源，因为它不利于：

- 稳定解析
- 搜索
- 回放
- 结构化落库

---

## 6. 平台记录映射

### 6.1 消息层

MVP 只把以下内容作为消息层：

- 成员输入
- Agent 最终回复

这也是搜索层唯一索引来源。

### 6.2 事件层

MVP 回放只需要以下核心事件：

- 状态变化
- 命令过程摘要
- 排队状态
- 可见性切换

建议最少支持以下事件类型：

- `status.changed`
- `command.summary`
- `message.queued`
- `session.shared`
- `session.privatized`

### 6.3 搜索口径

搜索必须严格遵守以下范围：

- 成员消息
- Agent 最终回复

明确不进入搜索的内容：

- 命令执行过程
- 状态事件
- 文件变化
- 其他过程事件

### 6.4 回放口径

回放必须严格遵守以下范围：

- 消息
- 状态变化
- 命令过程摘要

这里的命令过程摘要应面向人类阅读，而不是要求完整还原底层全部原始输出。

---

## 7. 调度与排队设计

### 7.1 为什么必须有调度器

产品允许多个成员连续往同一个共享会话发消息，但底层 Codex 会话更适合串行执行。

因此必须引入调度器，负责：

- 接收输入
- 维护每会话队列
- 维护每会话执行锁
- 在前端同步运行态与排队态

### 7.2 调度规则

建议固定如下：

- 一个 SyncAI 会话对应一个底层 Codex 会话
- 一个底层 Codex 会话同一时刻只处理一条执行任务
- 后续成员消息进入队列
- 队列按进入顺序消费
- 当前任务完成后再处理下一条

### 7.3 对前端暴露的状态

MVP 至少需要这些状态：

- `idle`
- `queued`
- `running`
- `completed`
- `error`

其中：

- `queued` 用于表示消息已入队但尚未执行
- `running` 用于表示当前会话正由 Codex 执行

---

## 8. Adapter 事件映射

为了让平台层稳定消费，`CodexAdapter` 输出必须先归一化，再进入业务层。

建议映射关系如下：

| 底层信号 | 平台记录类型 | 主要用途 |
|------|------|------|
| 会话启动 / 恢复 | `status.changed` | 更新会话运行态 |
| 正在执行 | `status.changed` | 前端展示运行中 |
| 新消息入队 | `message.queued` | 前端展示排队中 |
| 一段命令过程已完成整理 | `command.summary` | 供回放展示 |
| 最终回复产出 | `message(agent, is_final_reply=true)` | 会话主消息流、搜索 |
| 共享 / 私有切换 | `session.shared` / `session.privatized` | 审计与回放 |
| 底层失败 | `status.changed(error)` + 错误摘要 | 失败反馈与恢复 |

原则只有一个：
**平台依赖的是归一化后的结构，不依赖某个底层 CLI 的临时输出样式。**

---

## 9. 数据落库策略

### 9.1 必须落库的内容

MVP 建议至少持久化：

- 成员消息
- Agent 最终回复
- 状态变化事件
- 命令过程摘要事件
- 排队状态事件
- 共享 / 私有切换审计记录
- todo 与来源消息关系

### 9.2 为什么必须落库

因为 SyncAI 的价值不只在实时协作，还在沉淀：

- 搜索
- 回放
- 新成员接手
- 协作追踪
- 审计留痕

如果只做实时流，不做持久化，就无法形成团队资产。

---

## 10. 异常与恢复约束

MVP 不追求复杂的自动容灾，但至少要定义以下最小恢复规则：

- 底层 Codex 执行失败时，平台要把会话状态更新为 `error`
- 失败时要保留可阅读的错误摘要，而不是只保留一条黑盒失败
- 已入队但未执行的消息不能丢失
- 会话恢复后，允许继续消费后续队列或由成员再次发起输入
- 共享转私有的审计记录不能因为执行失败而丢失

---

## 11. 共享转私有的接入要求

共享转私有不是简单改一个可见性字段，接入层和业务层都要配合。

MVP 必须保证：

- 转换后，对其他成员隐藏整个会话
- 平台保留共享起止时间
- 平台保留当时可见成员范围
- 审计记录可用于证明该会话历史上曾被共享

也就是说：

**“转回私有”只改变后续可见性，不抹掉历史暴露事实。**

---

## 12. MVP 推荐实现方案

当前最适合的方案是：

1. 管理员本机运行 Codex
2. SyncAI 通过 `CodexAdapter` 接入该执行环境
3. 调度器按会话维度串行投递消息
4. Adapter 把状态、命令摘要、最终回复回传平台
5. 平台分别写入消息、事件、审计记录
6. WebSocket 向在线成员同步展示
7. 搜索只索引消息层
8. 回放只读取消息、状态变化、命令过程摘要

这条路径满足当前所有已拍板决策，而且没有引入额外未确认能力。

---

## 13. 当前是否可作为开发基线

可以。

这份接入设计已经明确了：

- 只接 Codex
- 单管理节点
- 单会话单底层会话
- 运行中消息入队
- 搜索范围
- 回放范围
- Adapter 生命周期
- 事件归一化规则
- 共享转私有审计要求

在此基础上，下一步应该进入接口定义、落库结构、测试脚本设计和最小可运行链路开发。
