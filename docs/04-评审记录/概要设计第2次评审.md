## 2. A组问题收口状态

### A-1 会话“创建即绑定”规则

- **收口状态：已关闭**
- **本轮已完成：**
- `docs/02-架构设计/概要设计.md` 已明确：会话创建前先读取团队当前 Codex 节点，先完成底层会话创建，再写入 `bound_agent_node_id` / `bound_agent_session_ref`；绑定失败则整次创建失败。
- `docs/02-架构设计/API设计.md` 已明确：`POST /projects/{project_id}/sessions` 成功响应必须返回 `bound_agent_node_id` 与 `bound_agent_session_ref`，且未配置节点/节点不可用时直接失败，不允许先建空壳会话。
- `docs/02-架构设计/数据库设计.md` 已改为正式口径：`bound_agent_node_id`、`bound_agent_session_ref` 为创建成功时的必备字段，并要求先拿到底层会话引用后再写入 `sessions`。
- `docs/03-开发与测试/开发任务拆解.md` 已前移节点配置与会话创建基础接口，并要求 Mock / Real Adapter 都在会话创建阶段返回 `bound_agent_session_ref`。
- **本轮补齐：**
  - `server/migrations/0001_initial_schema.sql` 已将 `bound_agent_node_id`、`bound_agent_session_ref` 收紧为 `NOT NULL`，与“创建成功即绑定”正式口径一致；
  - 至此“正式设计口径”和“首版落地基线”已闭环。

### A-2 “最终回复”是消息还是事件

- **收口状态：已关闭**
- **统一口径：**
  - 最终回复统一定义为 `agent message + is_final_reply = true`。
  - 它进入主消息流与搜索，不写成独立 `session_events.message.final`。
  - Adapter 内部输出合同已改为 `final.reply`，并显式注明这是“写入消息层”的信号，不是 `session_event_type`。
- **与现有落地稿的关系：**
  - `packages/shared/src/index.ts` 与 `server/migrations/0001_initial_schema.sql` 本来就没有 `message.final`，本轮文档已与该落地事实重新对齐。

### A-3 可见性切换是否进入回放时间轴

- **收口状态：已关闭**
- **统一口径：**
  - 回放时间轴现在正式包含 4 类条目：`message`、`status_changed`、`command_summary`、`visibility_changed`。
  - `session.shared` / `session.privatized` 既保留在 `session_events` 中供回放装配，也继续保留在 `session_audit_logs` 中承担审计职责。
  - WebSocket 的 `session.event.created` 已明确可推送 `session.shared`、`session.privatized` 这类可回放事件。
- **边界说明：**
  - 这不改变搜索边界；可见性切换仍不进入搜索。
  - 共享转私有后，对无权限成员仍隐藏整段会话内容；时间线节点只对当前仍有权限查看该会话的用户可见。
