## 2. 必须在第2次审查前补齐的问题

### A-1 缺少“私有 -> 共享”方向的显式测试用例

- **严重度：高**
- **问题说明：**
- `docs/01-产品/产品需求文档.md` 已明确“MVP 允许共享会话与私有会话互相转换”。
- 当前 `docs/03-开发与测试/测试用例设计.md` 的 F 组只覆盖了“共享 -> 私有”，没有覆盖“私有 -> 共享”。
- **当前证据：**
  - 已覆盖：
    - TC-F01 ~ TC-F05 均围绕共享转私有展开
  - 未覆盖：
    - 私有转共享后会话列表、详情、回放、搜索可见性恢复
    - `session.shared` 回放节点
- **结论：**
  - 这不是新的需求分歧，而是现有正式边界的一侧尚未被显式转成测试 case。

### A-2 缺少消息发送幂等与 `MESSAGE_DUPLICATED` 的显式测试用例

- **严重度：高**
- **问题说明：**
- `docs/02-架构设计/API设计.md` 已把 `client_message_id` 幂等与 `MESSAGE_DUPLICATED` 错误码写成正式接口合同。
- `docs/02-架构设计/数据库设计.md` 也已将 `(session_id, client_message_id)` 作为唯一性约束的一部分。
- 当前 `docs/03-开发与测试/测试用例设计.md` 没有任何 case 明确验证重复提交时的去重或拒绝语义。
- **当前证据：**
  - 正式合同存在：
- `docs/02-架构设计/API设计.md` 第 2.4 节
- `docs/02-架构设计/API设计.md` 错误码表中的 `MESSAGE_DUPLICATED`
  - 当前用例集中未出现：
    - `client_message_id`
    - `MESSAGE_DUPLICATED`
- **结论：**
  - 这是实时协作场景下的正式合同，不应留到脚本阶段才临时补想。

## 3. 建议在第2次审查前补齐的问题

### B-1 审计日志接口的读取路径还不够显式

- **严重度：中**
- **问题说明：**
  - 当前 TC-F02 已覆盖“存在完整审计记录”，但没有明确写清是通过 `GET /sessions/{session_id}/audit-logs` 合同层读取验证，还是仅通过数据库侧观察。
  - 对后续测试脚本设计来说，建议把接口入口写清，以免后续脚本层再二次猜测验证路径。

### B-2 列表 / 搜索分页与 cursor 合同尚未显式进入用例

- **严重度：中**
- **问题说明：**
- `docs/02-架构设计/API设计.md` 已统一规定列表接口优先使用 cursor 分页。
  - 当前测试用例重点放在核心闭环，这个取舍可以接受，但建议在第 2 次审查前至少补一条合同层分页 case，避免后续分页合同没有任何测试落点。
