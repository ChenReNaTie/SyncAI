## 1. 设计目标

当前测试脚本设计阶段只解决一个问题：

**如何把已通过两轮审查的 40 条测试用例，拆成可持续执行、可复用、可分层回归的测试脚本体系。**

本阶段必须完成以下 5 件事：

- 明确脚本分层，区分合同层、集成层、端到端层和人工补充验收层。
- 明确脚本执行顺序，保证从最小闭环到高成本协作验收逐层推进。
- 明确 40 条测试用例与脚本组之间的覆盖关系，避免漏测或重复堆叠。
- 明确夹具、数据准备、替身和环境隔离方式，保证脚本可重复执行。
- 明确脚本通过标准，让后续测试脚本审查与 TDD 开发前门禁有统一依据。

## 3. 脚本分层

### 3.1 合同层脚本

目标：

- 验证 HTTP / WebSocket / 回放条目 / 错误码 / 分页 / 幂等等正式接口合同是否稳定。

主要承接风险：

- 返回字段缺失或漂移
- 错误码与错误语义不一致
- 分页、幂等、审计读取路径未真正按接口合同兑现
- 消息层 / 事件层 / 审计层边界混淆

适合覆盖的重点用例：

- A 组中的创建、拦截、分页合同
- B 组中的消息发送幂等合同
- C / D / E / F 组中的回放、搜索、todo、审计接口合同

### 3.2 集成层脚本

目标：

- 验证 PostgreSQL、Redis、调度器、可见性过滤、Mock Adapter 归一化之间的联动正确性。

主要承接风险：

- 创建即绑定事务不完整
- 队列丢失或乱序
- 状态流转错误
- 最终回复、事件、审计记录持久化边界错误
- 共享 / 私有切换后的统一失效或统一恢复失败
- 服务重启后的恢复扫描失效

适合覆盖的重点用例：

- B / C / F / G 组绝大多数状态型与联动型用例

### 3.3 端到端脚本

目标：

- 从成员协作视角验证“可见、可协作、可回看、可沉淀”的最终产品效果。

主要承接风险：

- 多端实时状态不同步
- 列表 / 搜索 / 回放 / todo 入口割裂
- 可见性切换后页面状态与后端权限不同步
- 排队中 / 运行中交互体验不可信

适合覆盖的重点用例：

- A03、B06、D04、D05、E03、F01、F05、F06 等需要真实用户视角观察的场景

### 3.4 回归层脚本

目标：

- 把合同层、集成层、端到端层中已经稳定的脚本重新编排成提交前、日常回归、预发布回归三档固定回归集。

主要承接风险：

- 脚本虽已存在，但后续开发阶段没有稳定回归入口
- P0 / P1 优先级在脚本执行时被打散
- 发布前缺少一套可重复运行的全量回归编排

适合覆盖的重点用例：

- 提交前最小回归集中的全部 P0 用例
- 预发布前的全部 P0 + P1 自动化脚本
- 真实 Codex 补充验收前的环境健康检查与数据重置

### 3.5 人工补充验收

目标：

- 补齐当前仍不适合完全脚本化、但对产品价值判断很关键的结果。

主要承接风险：

- 命令摘要虽被正确记录，但人类不可读
- 错误摘要虽落库，但不足以让成员理解和继续协作
- 回放时间线虽排序正确，但整体可读性差

说明：

- 这一层不是替代自动化脚本，而是补充真实 Codex 环境下的产品感知验证。

## 4. 脚本组与覆盖关系

### 4.1 覆盖分组总览

| 脚本组 | 层级 | 目标 | 对应用例 |
|---|---|---|---|
| `contract_session_create_visibility` | 合同层 | 覆盖会话创建、会话详情 / 列表可见性、节点缺失 / 不可用失败、列表分页合同 | TC-A01、TC-A02、TC-A04、TC-A05、TC-A06、TC-A07 |

| `contract_replay_scope` | 合同层 | 覆盖回放条目类型、可见性切换条目、最终回复不写 `message.final` 事件 | TC-C04、TC-C06、TC-F03、TC-F07 |
| `contract_search_scope` | 合同层 | 覆盖搜索来源边界、可见性过滤、搜索分页合同 | TC-D01、TC-D02、TC-D03、TC-D04、TC-D05 |
| `contract_todo_and_audit` | 合同层 | 覆盖 todo 合同、审计日志接口字段与读取路径 | TC-E01、TC-E02、TC-F02 |
| `integration_session_binding_visibility` | 集成层 | 覆盖创建即绑定、共享 / 私有双向切换、审计 + 回放 + 可见性联动 | TC-A01、TC-A02、TC-A03、TC-A04、TC-A05、TC-A06、TC-F01、TC-F02、TC-F03、TC-F04、TC-F05、TC-F06、TC-F07 |

| `integration_execution_records` | 集成层 | 覆盖最终回复、状态事件、命令摘要、回放排序、消息层 / 事件层边界 | TC-C01、TC-C02、TC-C03、TC-C04、TC-C05、TC-C06 |
| `integration_failure_recovery` | 集成层 | 覆盖错误态、队列保留、发送阶段节点失败、恢复继续协作、重启恢复扫描 | TC-G01、TC-G02、TC-G03、TC-G04、TC-G05 |
| `e2e_shared_collaboration` | 端到端 | 覆盖共享会话历史、多成员协作、排队实时同步、最终回复与回放可读入口 | TC-A03、TC-B06、TC-C05 |
| `e2e_visibility_search_todo` | 端到端 | 覆盖私有访问拦截、搜索跳转、todo 来源跳转、共享 / 私有切换后的页面行为 | TC-A04、TC-D04、TC-D05、TC-E03、TC-F01、TC-F05、TC-F06 |
| `regression_commit_gate` | 回归层 | 提交前最小回归集，覆盖所有 P0 自动化关键用例 | TC-A01、TC-A05、TC-A06、TC-B01、TC-B02、TC-B03、TC-B07、TC-C01、TC-C03、TC-C04、TC-C06、TC-D01、TC-D03、TC-E01、TC-F01、TC-F02、TC-F03、TC-G01、TC-G02 |
| `regression_pre_release` | 回归层 | 预发布全量自动化回归，覆盖所有合同层 / 集成层 / 端到端脚本 | TC-A01 ~ TC-G05（自动化可承接部分） |
| `manual_real_codex_acceptance` | 人工补充验收 | 覆盖真实 Codex 下命令摘要可读性、错误摘要质量、回放整体可读性 | TC-C03、TC-C05、TC-G01、TC-G04 |

### 4.2 覆盖原则

- 每条测试用例必须至少有 1 个主脚本组承接。
- P0 用例允许同时被合同层和集成层双重覆盖，但必须有明确主归属，避免后续重复实现。
- 端到端脚本只承接“用户能否真正看到 / 操作到”的验证，不重复承担数据库约束的精确断言。
- 回归层不引入新的断言类型，只负责把已存在脚本组重新编排成固定回归入口。
- 人工补充验收只用于补充真实产品感知，不替代自动化回归集。

## 5. 夹具与数据准备

### 5.1 基础环境夹具

每次脚本执行前，至少准备：

- 1 个 PostgreSQL 测试库或独立 schema
- 1 个 Redis 独立 DB 或独立 key 前缀
- 1 个后端应用实例
- 1 个 WebSocket 可连接入口
- 1 套可切换的 Adapter 运行模式：`mock` / `real`

### 5.2 账号与组织夹具

建议固定以下基础账号：

- `admin_a`
- `member_a`
- `member_b`

建议固定以下组织夹具：

- `team_default`
- `project_default`
- `team_without_node`
- `team_with_online_node`
- `team_with_offline_node`

### 5.3 会话夹具

建议至少准备以下会话态：

- `session_shared_idle`
- `session_private_idle`
- `session_shared_with_history`
- `session_private_with_history`
- `session_running_with_queue`
- `session_error_recoverable`
- `session_privatized_history`
- `session_shared_again_history`

### 5.4 消息与事件夹具

建议准备以下消息 / 事件输入模板：

- `message_member_unique_keyword`
- `message_agent_final_unique_keyword`
- `message_duplicate_client_id_pair`
- `message_for_todo_source`
- `message_failure_with_error_summary`
- `event_status_running`
- `event_status_error`
- `event_message_queued`
- `event_command_summary`
- `event_visibility_shared`
- `event_visibility_privatized`

### 5.5 审计与时间控制夹具

建议准备：

- `audit_shared_to_private`
- `audit_private_to_shared`
- `visible_scope_snapshot_team_members`
- 可控时钟或固定时间戳生成器，用于稳定验证：
  - `shared_started_at`
  - `shared_ended_at`
  - `messages.created_at`
  - `session_events.occurred_at`

### 5.6 数据准备与清理策略

- 合同层脚本可按“每文件一组最小夹具”准备数据，避免跨文件耦合。
- 集成层脚本必须保证：
  - 每组执行前清空会话队列
  - 每组执行后清理消息 / 事件 / 审计脏数据
  - 不复用上组遗留的运行态缓存
- 端到端脚本优先使用新建团队 / 项目 / 会话，避免历史态串扰。
- 回归层执行前必须统一调用一次环境重置与健康检查，避免把上轮失败环境直接带入下轮结果。

## 7. 推荐目录骨架

当前阶段先定义推荐结构，不直接创建脚本目录。后续正式落地时建议采用：

```text
tests/
  contracts/
    session/
    message/
    replay/
    search/
    todo/
    audit/
  integration/
    binding/
    dispatch/
    execution/
    visibility/
    recovery/
  e2e/
    collaboration/
    search/
    todo/
    visibility/
  acceptance/
    codex/
  regression/
    commit-gate/
    pre-release/
  fixtures/
    users/
    teams/
    projects/
    sessions/
    messages/
    events/
    audits/
  mocks/
    codex/
    websocket/
    clock/
  helpers/
    db/
    redis/
    auth/
```

目录原则：

- 按测试分层与能力拆目录，不按页面堆叠。
- `fixtures/` 只承载稳定复用数据准备，不掺杂断言逻辑。
- `mocks/codex` 单独维护 Mock Adapter 模板，避免散落在各脚本中。
- `helpers/` 只放通用驱动、登录、订阅、重置工具，不放具体场景断言。
- `regression/` 只做脚本编排和套件聚合，不重复实现底层断言。

## 8. 失败后如何定位与回放

### 8.1 失败定位顺序

任一脚本失败后，统一按以下顺序定位：

1. **先看脚本层级**
   - 合同层失败：优先看 HTTP 响应、错误码、字段缺失、WebSocket 事件类型
   - 集成层失败：优先看数据库持久化、Redis 队列、状态流转、Adapter mock 输出
   - 端到端失败：优先看页面状态、网络请求、WebSocket 顺序、权限切换后的 UI 行为
   - 回归层失败：优先定位是底层脚本失败，还是回归编排、环境准备、套件顺序问题
2. **再看失败类型**
   - 需求偏差：现象与 `docs/产品需求文档.md` 冲突
   - 设计偏差：现象与 `docs/概要设计.md` / `docs/API设计.md` / `docs/数据库设计.md` 冲突
   - 实现缺陷：设计正确，但代码、SQL、调度、前端渲染没有兑现
   - 环境阻塞：数据库、Redis、Mock Adapter、真实 Codex 环境不可用
3. **最后看是否能稳定复现**
   - 若可稳定复现，直接转入根因分析
   - 若不可稳定复现，先补齐时间戳、消息 ID、WebSocket 顺序与环境日志，再判定是否为时序问题

### 8.2 失败回放材料

每次失败至少要保留下列材料，供二次回放：

- 脚本组名称
- 用例 ID
- 执行环境：`mock` / `real`
- 请求参数与响应体
- 关键 `cursor`
- 关键 `client_message_id`
- 关键 `session_id` / `message_id` / `event_id` / `audit_log_id`
- WebSocket 事件收包顺序
- PostgreSQL 快照查询结果
- Redis 队列与锁状态
- 若为端到端失败，再补页面截图、浏览器控制台输出、网络面板请求序列

### 8.3 回放方法

建议为每类失败准备固定回放入口：

- 合同层失败：
  - 复用同一最小夹具
  - 用同一请求体和同一 `client_message_id` 重放
- 集成层失败：
  - 用固定时钟、固定种子数据、固定 Mock Adapter 模板重放
  - 重放前清空 Redis 运行态，避免队列残留污染
- 端到端失败：
  - 重建同一团队 / 项目 / 会话夹具
  - 重新拉起双客户端订阅，复放相同消息序列
- 回归层失败：
  - 先拆回触发失败的底层脚本组单跑
  - 只有单跑也失败，才进入根因分析；若单跑通过，则优先排查回归编排顺序或环境重置缺失

### 8.4 根因归档要求

测试失败关闭前，必须至少补齐：

- 根因分类
- 首次失败时间
- 稳定复现步骤
- 修复后的复跑结果
- 若是设计偏差，回指受影响文档段落

## 9. 证据沉淀与断言模型

每个脚本执行结果至少要沉淀以下证据：

- 用例 ID
- 所属脚本组
- 执行环境：`mock` / `real`
- 执行结果：通过 / 失败 / 阻塞
- 核心证据：
  - HTTP 响应体
  - WebSocket 事件序列
  - `cursor`
  - `client_message_id`
  - 关键消息 ID
  - 关键事件 ID
  - 关键审计记录 ID
- 若失败，必须标记属于：
  - 需求偏差
  - 设计偏差
  - 实现缺陷
  - 环境阻塞

断言分三层：

- 接口层断言：状态码、错误码、字段、分页、幂等、条目类型
- 存储层断言：消息、事件、todo、审计记录、绑定字段、恢复扫描结果
- 实时层断言：运行中 / 排队中推送顺序、可见性切换后的订阅失效或恢复

### 10.1 测试脚本设计阶段通过标准

本阶段文档要达到“可进入测试脚本第 1 次审查”，至少同时满足：

- 已明确证据沉淀方式与通过标准

### 10.2 后续脚本执行阶段通过标准

- P0 用例在 `mock` 环境下必须全部通过

- 任一失败用例都必须能回指到具体证据与根因归类
