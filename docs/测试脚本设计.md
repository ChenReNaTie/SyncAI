# SyncAI / 灵悉 AI - 测试脚本设计

**版本：** v1.0  
**更新时间：** 2026-04-15  
**文档定位：** 基于 `docs/测试用例设计.md` 输出的自动化测试脚本设计文档，用于后续搭建测试目录、编写模拟器、组织接口测试和端到端测试。

---

## 1. 设计目标

测试脚本设计的目标不是现在就绑定具体测试框架，而是先确定脚本分层、数据夹具、模拟边界、执行顺序和命名规则，让后续代码初始化后可以直接落地。

本设计要保证：
- 能稳定覆盖 MVP 拍板口径。
- 能在“真实 Codex 环境”和“受控模拟适配器”之间切换。
- 能把串行调度、排队、回放、搜索、todo、审计拆成可重复执行的脚本组。
- 能支持后续 CI 或本地回归运行。

---

## 2. 脚本分层

### 2.1 合同层脚本

目标：验证服务边界和数据结构，不依赖完整页面。

建议覆盖：
- 会话创建返回结构。
- 消息发送返回结构。
- 回放接口返回的条目类型。
- 搜索结果只包含消息层记录。
- todo、审计记录的数据字段完整性。

适合承接用例：
- TC-A01、TC-A02、TC-C01、TC-C02、TC-C04、TC-D01、TC-D02、TC-D03、TC-E01、TC-E02、TC-F02。

### 2.2 集成层脚本

目标：验证数据库、Redis、调度服务、CodexAdapter 归一化之间的联动。

建议覆盖：
- 单会话执行锁。
- FIFO 队列消费。
- 消息状态流转。
- 失败后队列保留。
- 审计记录与可见性切换联动。

适合承接用例：
- TC-B01、TC-B02、TC-B03、TC-B04、TC-B05、TC-C03、TC-F01、TC-F03、TC-G01、TC-G02、TC-G03、TC-G04。

### 2.3 端到端脚本

目标：从成员视角验证页面和实时协作体验。

建议覆盖：
- 多成员同时打开同一共享会话。
- 一端发送消息，另一端看到运行中和排队中。
- todo 侧边栏提取与跳转。
- 共享转私有后其他成员页面立即失效或消失。
- 搜索跳转和回放展示结果。

适合承接用例：
- TC-A03、TC-A04、TC-B06、TC-C05、TC-D04、TC-E03、TC-F04。

### 2.4 手工补充验证

目标：验证真实 Codex 环境下难以完全脚本化的表现。

建议保留：
- 命令过程摘要的人类可读性验收。
- 错误摘要是否足够让成员理解并继续协作。
- 回放时间线是否满足产品展示预期。

## 3. 测试替身与模拟策略

### 3.1 CodexAdapter 双模式

建议为 Adapter 准备两种运行模式：
- `mock`：由脚本精确控制状态变化、命令摘要、最终回复、失败注入。
- `real`：连接真实 Codex 环境，用于闭环联调和验收。

`mock` 模式至少要支持以下输入模板：
- 成功执行并返回最终回复。
- 成功执行并返回多段命令摘要。
- 长执行，便于制造排队场景。
- 执行中失败。
- 节点不可用。

### 3.2 WebSocket 测试客户端

建议准备多客户端测试助手，用于：
- 模拟两个成员同时订阅同一会话。
- 记录消息、状态、排队事件的接收顺序。
- 校验不同成员在可见性切换后的推送差异。

### 3.3 数据库与缓存隔离

每轮自动化执行应满足：
- 使用独立测试库或独立 schema。
- Redis 使用独立前缀或独立 DB。
- 每个测试组有可重置夹具，不依赖上一个测试组遗留数据。

## 4. 脚本分组设计

### 4.1 `session_lifecycle`
- 目标：覆盖团队、项目、共享会话、私有会话的创建和可见性读取。
- 对应用例：TC-A01、TC-A02、TC-A03、TC-A04。
- 关键断言：会话绑定 Codex 引用正确、列表可见性正确、私有访问被拦截。

### 4.2 `dispatch_queue`
- 目标：覆盖消息接收、串行调度、排队和 FIFO 消费。
- 对应用例：TC-B01 ~ TC-B06。
- 关键断言：同一会话不存在双 `running`，排队顺序稳定，实时状态推送一致。

### 4.3 `execution_records`
- 目标：覆盖最终回复、状态事件、命令摘要的沉淀规则。
- 对应用例：TC-C01 ~ TC-C05。
- 关键断言：消息层和事件层边界稳定，回放可读且顺序正确。

### 4.4 `search_scope`
- 目标：覆盖搜索命中边界和可见性过滤。
- 对应用例：TC-D01 ~ TC-D04。
- 关键断言：只搜索消息层，不泄露私有会话内容。

### 4.5 `todo_sidebar`
- 目标：覆盖轻量 todo 的创建、状态切换、来源跳转。
- 对应用例：TC-E01 ~ TC-E03。
- 关键断言：字段极简、状态受限、来源跳转稳定。

### 4.6 `visibility_audit`
- 目标：覆盖共享转私有后的隐藏、审计、搜索失效。
- 对应用例：TC-F01 ~ TC-F04。
- 关键断言：内容不可见和审计留痕同时成立。

### 4.7 `failure_recovery`
- 目标：覆盖失败、节点不可用、恢复继续协作。
- 对应用例：TC-G01 ~ TC-G04。
- 关键断言：错误可读、队列不丢、恢复后可继续协作。

## 6. 断言模型

每类脚本都应至少校验三层结果：
- 接口层：返回码、返回体、字段合法性。
- 存储层：数据库中的消息、事件、todo、审计记录是否正确。
- 实时层：WebSocket 推送的内容、顺序、可见性是否正确。

关键断言清单：
- 会话状态和消息状态流转是否符合定义枚举。
- `command.summary` 是否只进入事件层。
- `is_final_reply=true` 的 Agent 消息是否进入消息层。
- 搜索是否完全忽略事件层。
- 共享转私有后，其他成员的搜索、列表、详情是否全部失效。
- 执行失败后，队列中未消费消息是否保留。

当前仓库还没有正式代码骨架，因此本轮先不直接创建测试目录，只定义推荐结构。待代码初始化后，建议落成如下目录：

- `mocks/codex` 独立承载受控适配器替身。

## 8. 执行顺序建议

### 8.1 本地开发回归顺序

1. `session_lifecycle`
2. `dispatch_queue`
3. `execution_records`
4. `search_scope`
5. `todo_sidebar`
6. `visibility_audit`
7. `failure_recovery`

### 8.2 提交前最小回归集

建议至少跑通：
- 创建共享会话。
- 空闲会话消息立即执行。
- 运行中消息进入队列。
- 最终回复写入消息层。
- 搜索不命中命令摘要。
- 手动提取 todo。
- 共享转私有后隐藏会话。
- 执行失败后保留队列。

### 8.3 联调验收集

在真实 Codex 环境下，至少补跑：
- 多成员共享会话协作。
- 命令过程摘要可读性。
- 回放时间线完整性。
- 单节点失败后的恢复协作。

## 9. 脚本命名建议

建议统一命名格式：
- 合同层：`contract_<能力>_<场景>`
- 集成层：`integration_<能力>_<场景>`
- 端到端：`e2e_<能力>_<场景>`

示例：
- `contract_session_create_shared`
- `integration_dispatch_fifo_queue`
- `e2e_visibility_privatize_hide_session`

这样可以直接从文件名判断测试目标，并与用例编号一一映射。
