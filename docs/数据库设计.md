# SyncAI / 灵悉 AI - 数据库设计

**版本：** v1.1  
**更新时间：** 2026-04-16  
**文档定位：** 基于 `docs/概要设计.md` 输出的正式数据库设计文档，作为 PostgreSQL 建表、迁移、索引、搜索、回放与恢复实现的直接基线。

---

## 1. 设计目标

本设计只服务当前 MVP，目标是把已拍板能力落实为可直接开发表结构和存储规则。

本设计必须满足以下约束：

- 组织结构固定为团队 > 项目 > 会话。
- 第一阶段只支持 `Codex`。
- Agent 默认运行在管理员本机。
- 同一会话按串行规则执行，运行中后续消息入队。
- 搜索第一版只覆盖成员消息和 Agent 最终回复。
- 回放第一版覆盖消息、状态变化、命令过程摘要和可见性切换节点。
- todo 只做轻量版：标题、状态、来源消息。
- 共享转私有后，对其他成员隐藏会话，但保留完整审计记录。
- 会话创建成功时必须立即绑定团队当前 Codex 节点，不允许先落一个未绑定 Session 再在首条消息发送时补绑。

## 2. 存储分层

| 存储 | 承担内容 | 不承担内容 |
|---|---|---|
| PostgreSQL | 业务主数据、消息、事件、todo、审计、会话绑定、可见性 | 临时执行锁、瞬时排队队列 |
| Redis | 每会话执行锁、排队队列、运行态缓存、节点心跳缓存 | 业务主数据最终真相来源 |

落地原则：

- PostgreSQL 是唯一主存。
- Redis 只承担运行态和调度态，不替代持久化数据。
- 所有成员消息必须先写 PostgreSQL，再进入 Redis 队列或执行。

## 3. 核心实体总览

| 表 / 结构 | 作用 | 关键关系 |
|---|---|---|
| `users` | 平台用户 | 1:N 到 `team_members`、`projects`、`sessions`、`messages`、`todos` |
| `teams` | 团队顶层容器 | 1:N 到 `team_members`、`projects`、`team_agent_nodes` |
| `team_members` | 团队成员关系 | N:1 到 `teams`、`users` |
| `team_agent_nodes` | 团队绑定的管理员本机 Codex 节点配置 | N:1 到 `teams`、`users` |
| `projects` | 项目容器 | N:1 到 `teams` |
| `sessions` | 共享 / 私有会话主表 | N:1 到 `projects`、`users`、`team_agent_nodes` |
| `messages` | 成员消息和 Agent 最终回复 | N:1 到 `sessions`、`users` |
| `session_events` | 状态变化、命令摘要、排队等结构化事件 | N:1 到 `sessions`，可选 N:1 到 `messages` |
| `todos` | 轻量待办 | N:1 到 `sessions`、`messages`、`users` |
| `session_audit_logs` | 共享 / 私有切换审计记录 | N:1 到 `sessions`、`users` |

## 4. 状态字典与枚举

- `idle`
- `queued`
- `running`
- `completed`

### 4.5 消息发送者与处理状态

- `sender_type`: `member` / `agent`
- `processing_status`: `accepted` / `queued` / `running` / `completed` / `failed`

### 5.7 `messages`

| 字段 | 类型 | 约束 | 说明 |
|---|---|---|---|
| `id` | `uuid` | PK | 消息 ID |
| `session_id` | `uuid` | FK -> `sessions.id` | 所属会话 |
| `sender_type` | `varchar(20)` | Not Null | `member` / `agent` |
| `sender_user_id` | `uuid` | FK -> `users.id`, Null | 成员消息时必填 |
| `content` | `text` | Not Null | 消息正文 |
| `content_format` | `varchar(20)` | Not Null, Default `markdown` | 当前建议 `markdown` |
| `processing_status` | `varchar(20)` | Not Null | 消息处理状态 |
| `is_final_reply` | `boolean` | Not Null, Default `false` | 仅 Agent 最终回复为 `true` |
| `sequence_no` | `bigint` | Not Null | 会话内单调递增序号 |
| `client_message_id` | `varchar(100)` | Null | 前端幂等键 |
| `error_summary` | `text` | Null | 当前消息失败时的可读摘要 |
| `metadata` | `jsonb` | Not Null, Default `{}` | 扩展字段 |
| `created_at` | `timestamptz` | Not Null | 创建时间 |
| `updated_at` | `timestamptz` | Not Null | 更新时间 |

搜索索引建议：

- 增加 `search_vector tsvector`
- 只为以下消息建搜索索引：
  - `sender_type = 'member'`
  - `sender_type = 'agent' AND is_final_reply = true`

示例：

```sql
CREATE INDEX idx_messages_search
ON messages
USING GIN (search_vector)
WHERE sender_type = 'member'
   OR (sender_type = 'agent' AND is_final_reply = TRUE);
```

## 6. 关键关系与约束规则

### 6.1 可见性规则

- `sessions.visibility = private` 时，只有 `creator_id` 本人可见。
- `sessions.visibility = shared` 时，默认项目所属团队成员可见。
- 搜索、回放、todo 查询必须统一套用会话可见性过滤。
- 共享 / 私有切换除审计表外，还必须在 `session_events` 中保留可回放事件。

### 6.2 串行执行规则

- 同一 `session_id` 任一时刻最多一条 `messages.processing_status = running`。
- 后续消息进入 `accepted -> queued -> running -> completed/failed` 流转。
- 失败不允许自动丢弃后续 `queued` 消息。

### 6.3 搜索范围规则

只允许以下记录进入搜索：

- 成员消息
- Agent 最终回复

明确不进入搜索：

- `session_events`
- 命令过程摘要
- 排队事件
- 可见性切换事件
- 其他过程事件

### 6.4 回放范围规则

回放由以下两层装配：

- `messages`
- `session_events`

回放只展示：

- 消息
- 状态变化
- 命令过程摘要
- 可见性切换

## 7. Redis 运行态设计

建议键设计如下：

| Key | 类型 | 作用 |
|---|---|---|
| `session:{session_id}:lock` | String | 会话执行锁 |
| `session:{session_id}:queue` | List / Stream | 待执行消息队列 |
| `session:{session_id}:runtime` | Hash | 当前运行态缓存 |
| `team:{team_id}:node:{node_id}:heartbeat` | Hash | 节点心跳与连接态 |

注意：

- Redis 队列中只存 `message_id`，不重复存消息正文真值。
- 进程启动后应根据 PostgreSQL 中未完成消息做恢复扫描。

## 8. 事务与一致性规则

### 8.1 创建会话

创建成功前必须先完成：

1. 读取团队当前可用 `team_agent_nodes`
2. 调用 Adapter `startSession`
3. 拿到 `bound_agent_session_ref`

随后在一个事务内完成：

1. 写入 `sessions`
2. 写入 `bound_agent_node_id`
3. 写入 `bound_agent_session_ref`
4. 记录初始字段

若任一步失败，则不允许留下未绑定 `sessions` 记录。

### 8.2 发送成员消息

一个事务内完成：

1. 写入 `messages`
2. 更新 `sessions.last_message_at`
3. 根据当前会话状态更新消息初始状态

事务提交后再执行：

4. 写 Redis 队列或直接触发调度
5. 推送 WebSocket

### 8.3 写入执行结果

建议一个事务内完成：

1. 写入 `session_events(status.changed)`
2. 写入 `session_events(command.summary)`，若有
3. 写入 Agent 最终回复到 `messages`
4. 更新当前成员消息 `processing_status`
5. 更新 `sessions.runtime_status`

### 8.4 共享转私有

必须一个事务内同时完成：

1. 更新 `sessions.visibility`
2. 写入 `session_events(session.privatized)`
3. 写入 `session_audit_logs`

## 9. 搜索与回放落地建议

### 9.1 搜索

MVP 建议直接使用 PostgreSQL 全文搜索，不额外引入独立搜索引擎。

原因：

- 第一版搜索范围很窄，只搜消息层。
- 数据量初期可控。
- 可以减少基础设施复杂度。

### 9.2 回放

回放不单独建表，直接按时间序读取：

- `messages.created_at`
- `session_events.occurred_at`

组装时应统一排序字段，避免消息和事件时间轴错乱，并确保 `session.shared` / `session.privatized` 可以作为独立时间线节点返回。
