## 1. 测试设计目标

本阶段测试设计只服务当前已通过第 3 次评审的 SyncAI MVP，不扩展未拍板能力，不提前进入执行脚本实现。

测试设计需要回答 5 个问题：

- 当前 MVP 到底要验证哪些业务闭环。
- 测试应分成哪几层，每层分别承接什么风险。
- 哪些路径属于必须优先覆盖的核心业务链路。
- 哪些风险虽不是概要设计门禁，但必须在测试设计中显式覆盖。
- 什么状态才能算“测试设计阶段成型”，可以进入后续审查与脚本设计。

## 3. 测试设计原则

- **只测已拍板边界**：不把 Claude Code、多节点、复杂权限、AI 自动提 todo 等非 MVP 能力混入当前测试范围。
- **先闭环再扩展**：先证明“创建会话 -> 发送消息 -> 排队执行 -> 最终回复 -> 回放 / 搜索 / todo / 审计”主链路成立，再谈补充场景。
- **消息层与事件层分开验**：最终回复属于消息层；状态变化、命令摘要、可见性切换属于事件层；测试必须显式验证边界不会串层。
- **先设计可追溯性**：每条核心测试都必须能回指到概要设计、API 设计或数据库设计中的正式规则。
- **失败路径不能后置**：单节点不可用、创建即绑定失败、执行失败、恢复继续协作都属于 MVP 必测入口。

## 4. 测试分层

### 4.1 合同层测试

目标：

- 验证 HTTP / WebSocket / 内部归一化合同是否满足正式设计字段与边界要求。

重点覆盖：

- 会话创建返回 `bound_agent_node_id`、`bound_agent_session_ref`
- 消息发送返回状态
- 回放条目类型
- 搜索结果结构与范围
- todo / 审计 / 可见性切换接口字段

适合发现的问题：

- 字段缺失
- 返回类型漂移
- 错误码或错误语义不一致
- 消息层 / 事件层 / 审计层边界混淆

### 4.2 集成层测试

目标：

- 验证 PostgreSQL、Redis、调度器、Adapter 归一化和可见性过滤的联动正确性。

重点覆盖：

- 创建即绑定的事务闭环
- 单会话执行锁
- FIFO 队列消费
- 最终回复、事件、审计记录的持久化边界
- 共享转私有后的统一失效
- 失败后的队列保留与恢复

适合发现的问题：

- 状态流转错误
- 事务不完整
- 队列丢失 / 乱序
- 可见性过滤遗漏

### 4.3 端到端测试

目标：

- 从成员协作视角验证“可见、可协作、可回看、可沉淀”的产品效果。

重点覆盖：

- 多成员共享会话
- 一端运行中，另一端排队中可见
- 搜索跳转与回放时间线
- todo 提取与来源跳转
- 共享转私有后其他成员立即失效

适合发现的问题：

- 页面状态与后端状态不一致
- WebSocket 时序错误
- 列表 / 搜索 / 回放入口体验断裂

### 4.4 人工补充验收

目标：

- 覆盖难以完全脚本化、但对产品感知很关键的结果。

重点覆盖：

- 命令摘要的人类可读性
- 错误摘要是否足够让成员理解
- 回放时间线是否具备产品可读性

说明：

- 这一层是测试设计的一部分，但不是本轮要写的执行脚本。

## 5. 核心业务链路测试范围

### 5.1 链路一：会话创建即绑定

要验证：

- 共享 / 私有会话创建成功时，必须立即拿到 `bound_agent_node_id` 与 `bound_agent_session_ref`
- 团队未配置节点或节点不可用时，创建直接失败
- 失败时不得留下未绑定会话空壳

### 5.2 链路二：消息串行调度与排队

要验证：

- 空闲会话消息立即执行
- 运行中后续消息进入队列
- 同一会话任意时刻最多一条消息 `running`
- 队列严格按 FIFO 消费
- 多端可以实时看到运行中 / 排队中状态

### 5.3 链路三：执行结果沉淀

要验证：

- 最终回复进入消息层，并带 `is_final_reply = true`
- `status.changed`、`command.summary`、`message.queued` 进入事件层
- 不产生独立的 `message.final` 事件
- 搜索只从消息层取数

### 5.4 链路四：回放与搜索

要验证：

- 回放时间线包含消息、状态变化、命令摘要、可见性切换
- 回放不包含原始终端日志与未定义过程事件
- 搜索只命中成员消息和 Agent 最终回复
- 回放与搜索都统一受可见性过滤

### 5.5 链路五：todo 与共享转私有

要验证：

- todo 只能手动提取
- todo 只允许 `pending / completed`
- todo 能跳回来源消息
- 共享转私有后，对其他成员隐藏会话
- 平台同时保留回放时间线节点与审计记录

### 5.6 链路六：异常与最小恢复

要验证：

- 执行失败时消息 / 会话状态正确
- 错误摘要清晰可见
- 当前失败不会吞掉后续排队消息
- 节点不可用时创建或发送都会返回明确错误
- 恢复后允许继续协作

## 6. 关键风险覆盖点

基于概要设计第 3 次评审，本阶段必须显式覆盖以下风险：

- **R-1 单管理员节点可用性风险**
  - 覆盖点：未配置节点、节点不可用、恢复后继续协作
- **R-2 创建即绑定的原子性风险**
  - 覆盖点：创建失败不留空壳会话、绑定字段必填、错误码明确
- **R-3 消息层 / 事件层边界漂移风险**
  - 覆盖点：最终回复不是 `session_event`，命令摘要不进消息层
- **R-4 回放时间线排序风险**
  - 覆盖点：`messages.created_at` 与 `session_events.occurred_at` 混合排序正确
- **R-5 可见性切换泄漏风险**
  - 覆盖点：列表、详情、搜索、回放、todo、WebSocket 一致过滤
- **R-6 审计记录不完整风险**
  - 覆盖点：`previous_visibility`、`new_visibility`、`shared_started_at`、`shared_ended_at`、`visible_scope_snapshot`
- **R-7 当前起点描述过时风险**
  - 覆盖点：测试用例不依赖“仓库当前已有多少代码”，只依赖正式设计和最终应有行为

## 7. 通过标准

测试设计阶段要达到“成型可审查”，至少同时满足以下条件：

- 已有 1 份测试设计总文档，明确测试目标、分层、链路、风险、通过标准。
- 已有 1 份测试用例设计文档，按模块列出核心 case，并包含优先级。
- 核心业务链路 6 条全部有对应测试覆盖。
- 已拍板规则全部转成明确测试断言，不能停留在描述层。
- 不存在新的需求分歧、口径冲突或概要设计级门禁。
- 文档能直接作为“测试用例第 1 次审查”的输入。

说明：

- 当前阶段的通过，不等于测试已经执行，也不等于代码已经完成。
- 当前阶段只表示“测试设计已经成型，可以进入测试用例审查”。

## 8. 这一阶段最终应该长什么样

拿铁问“测试设计最终会是什么效果”，本阶段完成态应明确表现为：

- **效果 1：测试目标清楚**
  - 团队一眼能知道当前 MVP 到底要测哪些闭环、不测哪些范围。
- **效果 2：风险入口清楚**
  - 团队知道单节点失败、创建即绑定、回放边界、可见性泄漏等高风险点都被显式覆盖。
- **效果 3：用例列表清楚**
  - 每个模块都已有可审查的核心 case，后续不需要再回到概要设计重新猜边界。
- **效果 4：下一步衔接清楚**
  - 先做测试用例 2 次审查，再基于审查后的用例进入测试脚本 / 测试脚本设计，不直接跳到编码实现。

换句话说，本阶段结束后，项目手里应至少有两份可直接使用的正式产物：

1. `docs/03-开发与测试/测试设计.md`：说明“为什么测、测什么、按什么层测、什么叫过关”
2. `docs/03-开发与测试/测试用例设计.md`：说明“每个模块具体有哪些核心 case 要审、要验”

这就是当前测试设计阶段应达到的最终效果。
