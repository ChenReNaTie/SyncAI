# SyncAI / 灵悉 AI - 测试用例设计

**版本：** v1.0  
**更新时间：** 2026-04-15  
**文档定位：** 基于 `docs/概要设计.md` 输出的 MVP 测试用例文档，作为后续手工测试、接口联调、自动化脚本设计的直接输入。

---

## 1. 测试目标

本轮测试设计只验证当前已拍板 MVP 闭环，不扩展未确认能力。测试目标是证明以下能力成立：
- 团队成员能围绕同一 Codex 会话持续协作。
- 同一会话消息能串行执行，运行中消息能够入队且前端可见。
- 平台能正确沉淀消息、状态变化、命令过程摘要，并支撑搜索和回放。
- 轻量 todo 与共享转私有审计规则符合基线口径。
- 单管理员节点约束下，异常场景仍能保证最小可恢复性。

---

## 2. 测试范围

### 2.1 覆盖范围

- 团队 > 项目 > 会话 三层结构。
- 共享会话 / 私有会话的可见性规则。
- 成员消息发送、串行调度、排队状态、最终回复回传。
- 搜索、回放、todo、审计记录。
- 单管理员节点不可用、执行失败等异常路径。

### 2.2 不覆盖范围

- Claude Code 接入。
- 多节点调度、自动故障切换。
- IDE 插件、移动端、复杂权限模型。
- AI 自动提取 todo。
- 搜索命令原始过程、文件变化和其他过程事件。

---

## 3. 测试环境基线

建议最小测试环境如下：
- 1 个管理员账号。
- 2 个普通成员账号。
- 1 个团队，团队内至少 1 个项目。
- 1 套可用的 Codex 测试执行环境。
- PostgreSQL、Redis、WebSocket 服务可用。
- 可切换 CodexAdapter 为“真实适配器”或“受控模拟适配器”。

账号命名建议：
- `admin_a`
- `member_a`
- `member_b`

## 5. 详细测试用例

### A. 组织与会话基础

#### TC-A01 创建共享会话
- 对应设计：`docs/概要设计.md` 第 6、10 章。
- 前置条件：管理员已创建团队和项目，成员已加入项目。
- 步骤：
  1. `member_a` 在项目下创建共享会话。
  2. 查询会话详情。
  3. 查询项目会话列表。
- 预期结果：
  - 会话创建成功，`visibility = shared`。
  - 会话已绑定 `bound_agent_type = codex` 和有效会话引用。
  - 会话初始状态为 `idle`。
  - 项目成员可以在会话列表看到该会话。

#### TC-A02 创建私有会话
- 对应设计：`docs/概要设计.md` 第 6、14 章。
- 前置条件：同 TC-A01。
- 步骤：
  1. `member_a` 在项目下创建私有会话。
  2. `member_a` 查询自己的会话列表。
  3. `member_b` 查询同项目会话列表。
- 预期结果：
  - 会话创建成功，`visibility = private`。
  - `member_a` 可见该会话。
  - `member_b` 不可见该会话。

#### TC-A03 新成员查看共享会话历史
- 对应设计：`docs/概要设计.md` 第 7、14 章。
- 前置条件：已存在共享会话和若干历史消息；`member_b` 新加入项目。
- 步骤：
  1. `member_b` 进入项目。
  2. 打开已有共享会话。
  3. 查看消息流和回放入口。
- 预期结果：
  - `member_b` 能看到共享会话历史。
  - 历史内容包含消息和已沉淀的回放内容。

#### TC-A04 私有会话访问拦截
- 对应设计：`docs/概要设计.md` 第 14 章。
- 前置条件：`member_a` 已创建私有会话。
- 步骤：
  1. `member_b` 通过会话 ID 直接访问该会话详情。
  2. `member_b` 访问该会话回放和搜索跳转结果。
- 预期结果：
  - 所有访问都被拦截或返回不可见。
  - `member_b` 不能通过直接链接绕过私有可见性。

### B. 串行调度与排队

#### TC-B01 空闲会话消息立即执行
- 对应设计：`docs/概要设计.md` 第 7、9、10 章。
- 前置条件：共享会话处于 `idle`。
- 步骤：
  1. `member_a` 发送一条成员消息。
  2. 观察消息状态和会话状态。
- 预期结果：
  - 消息先落库再进入执行。
  - 消息状态从 `accepted` 进入 `running`。
  - 会话状态变为 `running`。
  - 前端出现运行中反馈。

#### TC-B02 运行中消息进入队列
- 对应设计：`docs/概要设计.md` 第 7、10 章。
- 前置条件：同一会话已有一条消息执行中。
- 步骤：
  1. `member_b` 在执行未结束前再发送一条消息。
  2. 观察前端状态。
  3. 查询消息处理状态和事件记录。
- 预期结果：
  - 新消息不打断当前执行。
  - 新消息状态为 `queued`。
  - 产生 `message.queued` 事件。
  - 前端显示该消息处于排队中。

#### TC-B03 多条排队消息按 FIFO 顺序消费
- 对应设计：`docs/概要设计.md` 第 7、10、13 章。
- 前置条件：会话中已有 1 条运行中消息。
- 步骤：
  1. 连续发送第 2、3、4 条成员消息。
  2. 等待执行完成并观察消费顺序。
- 预期结果：
  - 第 2、3、4 条消息依次进入队列。
  - 当前消息完成后，队列严格按入队顺序消费。
  - 不出现跳号执行和并行执行。

#### TC-B04 同会话不存在双运行态
- 对应设计：`docs/概要设计.md` 第 5、9、13 章。
- 前置条件：共享会话存在高并发发送场景。
- 步骤：
  1. 模拟多个成员几乎同时向同一会话发送消息。
  2. 采集该会话每条消息的状态流转。
- 预期结果：
  - 任意时刻最多只有一条消息处于 `running`。
  - 会话级执行锁生效。

#### TC-B05 运行结束后自动消费下一条队列消息
- 对应设计：`docs/概要设计.md` 第 9、10 章。
- 前置条件：会话中有 1 条运行中消息和至少 1 条排队消息。
- 步骤：
  1. 等待当前执行完成。
  2. 观察下一条消息状态流转。
- 预期结果：
  - 当前消息完成后，下一条排队消息自动进入 `running`。
  - 会话状态从 `completed`/聚合完成态快速转入下一轮 `running`。

#### TC-B06 排队状态实时同步给多端成员
- 对应设计：`docs/概要设计.md` 第 4、10 章。
- 前置条件：`member_a` 和 `member_b` 同时在线查看同一共享会话。
- 步骤：
  1. `member_a` 触发一条长执行消息。
  2. `member_b` 发送后续消息。
  3. 观察两端页面状态变化。
- 预期结果：
  - 两端都能看到运行中与排队中的最新状态。
  - 状态变化与消息顺序一致。

### C. 执行结果沉淀与回放

#### TC-C01 最终回复写入消息层
- 对应设计：`docs/概要设计.md` 第 8、10、12 章。
- 前置条件：任意消息成功执行完成。
- 步骤：
  1. 等待 Codex 返回最终回复。
  2. 查询会话消息列表。
- 预期结果：
  - 产生一条 `sender_type = agent` 且 `is_final_reply = true` 的消息。
  - 最终回复可在主消息流中查看。

#### TC-C02 状态变化写入事件层
- 对应设计：`docs/概要设计.md` 第 8、9、12 章。
- 前置条件：执行流程已发生状态变化。
- 步骤：
  1. 查询会话事件列表。
  2. 校验关键状态事件顺序。
- 预期结果：
  - 至少存在 `status.changed` 事件。
  - 事件顺序与实际执行顺序一致。

#### TC-C03 命令过程摘要只进入回放不进入消息流
- 对应设计：`docs/概要设计.md` 第 8、10、12 章。
- 前置条件：执行过程中产生可读命令摘要。
- 步骤：
  1. 查看主消息流。
  2. 查看回放时间轴。
  3. 查询消息层数据。
- 预期结果：
  - 主消息流不把命令摘要当作成员/Agent 消息展示。
  - 回放中可见命令过程摘要。
  - 消息层中不存在把命令摘要误写为搜索消息的记录。

#### TC-C04 回放时间轴范围正确
- 对应设计：`docs/概要设计.md` 第 8、10 章。
- 前置条件：会话已有消息、状态变化、命令摘要。
- 步骤：
  1. 打开回放页面或接口。
  2. 校验返回条目类型。
- 预期结果：
  - 回放只包含消息、状态变化、命令过程摘要。
  - 不包含未定义的原始过程日志或文件变化细节。

#### TC-C05 回放顺序与时间线一致
- 对应设计：`docs/概要设计.md` 第 10、13 章。
- 前置条件：执行至少经历一次排队和一次成功完成。
- 步骤：
  1. 拉取完整回放时间轴。
  2. 核对条目顺序。
- 预期结果：
  - 回放按持久化时间顺序排列。
  - 同一轮执行中的消息、状态事件、命令摘要、最终回复顺序可读且不混乱。

### D. 搜索边界

#### TC-D01 搜索命中成员消息
- 对应设计：`docs/概要设计.md` 第 8、10、11 章。
- 前置条件：成员消息中含唯一关键词。
- 步骤：
  1. 用该关键词搜索。
  2. 点击搜索结果跳转。
- 预期结果：
  - 搜索命中该成员消息。
  - 跳转到对应会话和消息定位。

#### TC-D02 搜索命中 Agent 最终回复
- 对应设计：`docs/概要设计.md` 第 8、10、11 章。
- 前置条件：Agent 最终回复中含唯一关键词。
- 步骤：
  1. 用该关键词搜索。
  2. 查看结果来源。
- 预期结果：
  - 搜索命中 Agent 最终回复。
  - 结果展示所属项目、会话和命中时间。

#### TC-D03 搜索不命中事件层内容
- 对应设计：`docs/概要设计.md` 第 8、13 章。
- 前置条件：命令摘要或状态事件中含唯一关键词，但消息层不含该关键词。
- 步骤：
  1. 用该关键词搜索。
- 预期结果：
  - 搜索无结果。
  - 证明搜索来源仅为消息层。

#### TC-D04 搜索遵守可见性过滤
- 对应设计：`docs/概要设计.md` 第 11、14 章。
- 前置条件：`member_a` 有私有会话，包含唯一关键词。
- 步骤：
  1. `member_a` 搜索该关键词。
  2. `member_b` 搜索同一关键词。
- 预期结果：
  - `member_a` 可以命中私有会话内容。
  - `member_b` 不能命中该内容。

### E. 轻量 todo

#### TC-E01 从消息手动提取 todo
- 对应设计：`docs/概要设计.md` 第 7、10 章。
- 前置条件：会话中存在一条可提取的成员消息。
- 步骤：
  1. 对该消息执行“提取为 todo”。
  2. 查看会话侧边栏。
- 预期结果：
  - 新建 todo 成功。
  - todo 包含标题、状态、来源消息引用。
  - 初始状态为 `pending`。

#### TC-E02 todo 状态只允许待办/已完成
- 对应设计：`docs/概要设计.md` 第 6、11 章。
- 前置条件：会话中已有 todo。
- 步骤：
  1. 将 todo 从 `pending` 改为 `completed`。
  2. 尝试写入未定义状态值。
- 预期结果：
  - 合法状态切换成功。
  - 非法状态被拒绝。

#### TC-E03 todo 跳转来源消息
- 对应设计：`docs/概要设计.md` 第 8、10 章。
- 前置条件：会话中已有 todo 且来源消息存在。
- 步骤：
  1. 点击 todo。
- 预期结果：
  - 页面跳回来源消息位置。
  - 若来源消息所在会话可见，跳转结果稳定可用。

### F. 共享转私有与审计

#### TC-F01 共享转私有后会话对其他成员隐藏
- 对应设计：`docs/概要设计.md` 第 7、10、14 章。
- 前置条件：共享会话中已有历史消息。
- 步骤：
  1. `member_a` 将共享会话转为私有。
  2. `member_b` 查询会话列表、详情、搜索结果。
- 预期结果：
  - `member_b` 不再看到该会话及其内容。
  - `member_a` 仍能在私有状态下访问该会话。

#### TC-F02 共享转私有生成完整审计记录
- 对应设计：`docs/概要设计.md` 第 8、10、14 章。
- 前置条件：执行 TC-F01。
- 步骤：
  1. 查询审计记录。
- 预期结果：
  - 记录包含共享开始时间、共享结束时间、可见成员范围快照、操作人。
  - 审计记录与会话绑定正确。

#### TC-F03 系统保留“曾共享过”的事实
- 对应设计：`docs/概要设计.md` 第 2、8、14 章。
- 前置条件：执行 TC-F01。
- 步骤：
  1. 查询会话审计历史。
  2. 查询会话当前属性和历史属性。
- 预期结果：
  - 当前会话虽然已私有，但系统仍能识别其曾经是共享会话。
  - 审计记录未被清空或覆盖。

#### TC-F04 转私有后历史搜索结果失效
- 对应设计：`docs/概要设计.md` 第 11、14 章。
- 前置条件：共享状态下 `member_b` 曾能搜索到该会话中的关键词。
- 步骤：
  1. 将会话转为私有。
  2. `member_b` 再次搜索原关键词。
- 预期结果：
  - `member_b` 再次搜索时不再命中该会话内容。
  - 原有历史可见事实只保留在审计层，不继续暴露内容。

### G. 异常与最小恢复

#### TC-G01 Codex 执行失败时状态变为 error
- 对应设计：`docs/概要设计.md` 第 12、13 章。
- 前置条件：模拟 CodexAdapter 返回失败。
- 步骤：
  1. 发送成员消息。
  2. 注入执行失败。
  3. 查看消息状态、会话状态和错误摘要。
- 预期结果：
  - 当前消息进入 `failed`。
  - 会话状态进入 `error`。
  - 平台保留可读错误摘要。

#### TC-G02 当前执行失败不丢失后续排队消息
- 对应设计：`docs/概要设计.md` 第 13 章。
- 前置条件：会话中有 1 条运行消息和 2 条排队消息。
- 步骤：
  1. 让当前运行消息失败。
  2. 查询队列和消息状态。
- 预期结果：
  - 后续排队消息仍然存在。
  - 消息不会被误标记为已完成或被删除。

#### TC-G03 节点不可用时返回明确错误
- 对应设计：`docs/概要设计.md` 第 13 章。
- 前置条件：模拟管理员执行节点不可用。
- 步骤：
  1. 在会话中发送成员消息。
  2. 查看返回结果与状态。
- 预期结果：
  - 系统返回执行节点不可用的明确错误。
  - 不出现假成功。
  - 会话进入可见错误态或保留失败结果。

#### TC-G04 恢复后允许继续协作
- 对应设计：`docs/概要设计.md` 第 10、13 章。
- 前置条件：会话曾因失败进入 `error`，节点随后恢复。
- 步骤：
  1. 恢复执行环境。
  2. 重新发送一条成员消息或恢复消费队列。
- 预期结果：
  - 会话可再次进入 `queued`/`running`。
  - 协作链路恢复正常。
