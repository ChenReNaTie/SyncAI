## 1. 用例设计目标

本轮用例设计只覆盖当前已拍板的 SyncAI MVP，不进入执行脚本编写，不扩展未确认能力。

本文需要做到：

- 把概要设计中的核心业务闭环拆成可审查的测试 case。
- 让每条高风险规则都能找到对应断言入口。
- 为后续“测试用例 2 次审查”提供统一底稿。
- 为后续测试脚本设计保留稳定的用例编号与优先级结构。

## 4. 详细测试用例

### A. 会话创建与可见性基础

| 用例 ID | 名称 | 核心验证 | 前置条件 | 优先级 |
|---|---|---|---|---|
| TC-A01 | 创建共享会话成功即绑定 Codex | 创建共享会话成功时立即返回 `bound_agent_node_id`、`bound_agent_session_ref`，且 `runtime_status = idle` | 管理员已创建团队与项目，团队已配置可用节点 | P0 |
| TC-A02 | 创建私有会话成功即绑定 Codex | 私有会话同样遵守创建即绑定规则，且仅创建者可见 | 同 TC-A01 | P1 |
| TC-A03 | 新成员可见共享会话历史 | 新加入项目成员可看到共享会话中的消息流与回放入口 | 已有共享会话与历史消息 | P1 |
| TC-A04 | 私有会话访问被统一拦截 | 非创建者访问私有会话详情、回放、搜索跳转时全部被拦截 | 已存在私有会话 | P1 |
| TC-A05 | 未配置节点时会话创建直接失败 | `POST /sessions` 返回 `NODE_NOT_CONFIGURED`，且数据库中不存在空壳会话 | 团队未配置节点 | P0 |
| TC-A06 | 节点不可用或 `startSession` 失败时创建直接失败 | 返回 `NODE_UNAVAILABLE` 或等价错误，且不产生未绑定会话记录 | 团队已配置节点，但节点不可用或 Adapter 启动失败 | P0 |
| TC-A07 | 会话列表 cursor 分页稳定且不泄漏私有会话 | `GET /projects/{project_id}/sessions` 使用 `cursor` / `limit` 分页时，不重不漏，且每页都只返回当前成员有权看到的会话 | 项目下存在足够数量的共享 / 私有会话，能触发至少两页分页 | P1 |

### B. 串行调度与排队

| 用例 ID | 名称 | 核心验证 | 前置条件 | 优先级 |
|---|---|---|---|---|
| TC-B01 | 空闲会话消息立即执行 | 成员消息先落库，再进入 `running`，会话状态变为 `running` | 会话处于 `idle` | P0 |
| TC-B02 | 运行中消息进入队列 | 当前执行未结束时新增消息进入 `queued`，并产生 `message.queued` 事件 | 会话已有一条执行中消息 | P0 |
| TC-B03 | 多条排队消息按 FIFO 消费 | 队列严格按入队顺序消费，不跳号、不并发 | 会话已有 1 条运行中消息 | P0 |
| TC-B04 | 同会话不存在双运行态 | 任意时刻同一会话最多 1 条消息 `running` | 模拟多成员近同时发送消息 | P1 |
| TC-B05 | 当前执行结束后自动消费下一条队列消息 | 上一条完成后，下一条自动进入 `running` | 会话中有运行消息和至少 1 条排队消息 | P1 |
| TC-B06 | 排队状态实时同步给多端成员 | 多端同时看到运行中 / 排队中状态变化，且顺序一致 | 2 个成员同时订阅同一共享会话 | P1 |
| TC-B07 | 重复 `client_message_id` 不产生重复消息 | 同一会话内重复提交相同 `client_message_id` 时，返回 `MESSAGE_DUPLICATED` 或等价幂等拒绝语义，且不会重复写入消息、队列或事件 | 已有可发送成员消息的会话 | P0 |

### C. 执行结果沉淀与回放

| 用例 ID | 名称 | 核心验证 | 前置条件 | 优先级 |
|---|---|---|---|---|
| TC-C01 | 最终回复写入消息层 | 成功执行后生成 `sender_type = agent` 且 `is_final_reply = true` 的消息 | 任意消息成功执行完成 | P0 |
| TC-C02 | 状态变化写入事件层 | 关键执行状态变化产生 `status.changed` 事件，且顺序正确 | 执行流程已发生状态变化 | P1 |
| TC-C03 | 命令摘要只进入回放不进入消息流 | `command.summary` 可在回放中看到，但不进入主消息流与搜索消息层 | 执行过程中产生可读命令摘要 | P0 |
| TC-C04 | 回放范围包含可见性切换 | 回放只包含消息、状态变化、命令摘要、可见性切换，不包含原始终端日志 | 会话已有消息、事件、可见性切换记录 | P0 |
| TC-C05 | 回放顺序与时间线一致 | 消息、状态事件、命令摘要、可见性切换在时间轴中排序稳定可读 | 会话至少经历一次排队和一次可见性切换 | P1 |
| TC-C06 | 不生成独立 `message.final` 事件 | 最终回复以消息层落库，不额外写入 `session_events.message.final` | 任意消息成功执行完成 | P0 |

### D. 搜索边界

| 用例 ID | 名称 | 核心验证 | 前置条件 | 优先级 |
|---|---|---|---|---|
| TC-D01 | 搜索命中成员消息 | 成员消息中的唯一关键词可命中，并支持跳转到对应消息 | 成员消息中含唯一关键词 | P0 |
| TC-D02 | 搜索命中 Agent 最终回复 | Agent 最终回复中的唯一关键词可命中 | Agent 最终回复中含唯一关键词 | P1 |
| TC-D03 | 搜索不命中事件层内容 | 命令摘要、状态事件、可见性切换中的关键词不会被搜索命中 | 关键词只存在于事件层 | P0 |
| TC-D04 | 搜索遵守可见性过滤 | 私有会话或转私有后的会话内容不会继续被无权成员命中 | 已存在带唯一关键词的私有或已私有化会话 | P1 |
| TC-D05 | 搜索结果 cursor 分页稳定且继续遵守可见性过滤 | `GET /teams/{team_id}/search` 在 `cursor` / `limit` 分页下结果不重不漏，且后续分页不会把无权会话结果混入 | 搜索结果足以触发至少两页分页，且结果中同时存在共享与无权私有会话候选数据 | P1 |

### E. 轻量 todo

| 用例 ID | 名称 | 核心验证 | 前置条件 | 优先级 |
|---|---|---|---|---|
| TC-E01 | 从消息手动提取 todo | 新建 todo 成功，初始状态为 `pending`，并保留 `source_message_id` | 会话中存在可提取消息 | P0 |
| TC-E02 | todo 状态仅允许待办 / 已完成 | 合法状态切换成功，非法状态被拒绝 | 会话中已有 todo | P1 |
| TC-E03 | todo 可跳回来源消息 | 点击 todo 后能够回到来源消息位置 | 会话中已有 todo 且来源消息存在 | P1 |

### F. 可见性切换与审计

| 用例 ID | 名称 | 核心验证 | 前置条件 | 优先级 |
|---|---|---|---|---|
| TC-F01 | 共享转私有后对其他成员隐藏会话 | 其他成员在列表、详情中都不可再见到该会话 | 共享会话中已有历史消息 | P0 |
| TC-F02 | 共享转私有生成完整审计记录 | 通过 `GET /sessions/{session_id}/audit-logs` 读取到的审计记录包含 `previous_visibility`、`new_visibility`、`shared_started_at`、`shared_ended_at`、`visible_scope_snapshot`、`operator_id` | 执行一次共享转私有 | P0 |
| TC-F03 | 共享转私有写入回放时间线节点 | 回放中出现 `visibility_changed` 条目，且可读反映 `shared -> private` | 执行一次共享转私有 | P0 |
| TC-F04 | 系统保留“曾共享过”的事实 | 当前会话虽已私有，但系统仍可通过回放与审计证明其历史共享状态 | 执行一次共享转私有 | P1 |
| TC-F05 | 转私有后搜索 / 回放 / todo / WebSocket 统一失效 | 其他成员不再看到该会话搜索结果、回放内容、todo 和实时推送 | 成员曾在共享态下访问该会话 | P1 |
| TC-F06 | 私有转共享后项目成员重新获得可见性 | 会话从 `private -> shared` 后，项目成员重新可以在列表、详情、回放入口中看到该会话，并基于共享规则继续协作 | 已存在私有会话，且项目内还有其他成员 | P1 |
| TC-F07 | 私有转共享写入回放时间线与审计记录 | 回放中出现 `visibility_changed` 条目并可读反映 `private -> shared`，同时审计记录能通过 `GET /sessions/{session_id}/audit-logs` 读取到 `previous_visibility = private`、`new_visibility = shared` | 执行一次私有转共享 | P1 |

### G. 异常与最小恢复

| 用例 ID | 名称 | 核心验证 | 前置条件 | 优先级 |
|---|---|---|---|---|
| TC-G01 | 执行失败时消息与会话进入错误态 | 当前消息进入 `failed`，会话进入 `error`，并保留可读错误摘要 | 模拟 Adapter 返回执行失败 | P0 |
| TC-G02 | 当前执行失败不丢失后续排队消息 | 当前失败后，后续排队消息仍保留且状态正确 | 会话中有 1 条运行消息和至少 2 条排队消息 | P0 |
| TC-G03 | 发送阶段节点不可用时返回明确错误 | 发送消息时节点不可用，返回明确错误，不出现假成功 | 会话已绑定节点，但节点在发送阶段不可用 | P1 |
| TC-G04 | 恢复后允许继续协作 | 节点恢复后，会话可重新进入 `queued / running`，继续协作 | 会话曾进入 `error`，节点随后恢复 | P1 |
| TC-G05 | 服务重启后未完成状态可恢复扫描 | 重启后能根据持久化消息和事件恢复未完成状态，不依赖纯内存态 | 存在未完成消息或中断中的会话 | P1 |

## 6. 用例输出要求

后续每条用例无论人工执行还是脚本执行，至少要能沉淀以下结果：

- 用例 ID
- 执行环境
- 执行结果：通过 / 失败 / 阻塞
- 关键证据：接口响应、页面状态、消息 ID、事件 ID、审计记录 ID
- 若涉及分页或幂等，补充保留 `cursor`、`client_message_id` 与重复请求响应证据
- 若失败，明确属于需求偏差、设计偏差还是实现缺陷
