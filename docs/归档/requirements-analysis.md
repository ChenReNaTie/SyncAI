## 2. 产品目标梳理

### 2.1 产品一句话定义

**SyncAI（灵悉 AI）是一款面向团队的共享 AI 会话与协作跟进平台，把原本分散在个人设备和个人账号里的 AI 对话，转化为可共享、可追踪、可协同推进的团队资产。**

### 2.2 目标问题

产品要解决的核心问题有四类：

1. **对话记录孤岛**
   - 团队成员与 AI 的讨论仅保存在本地或个人账号中。
   - 其他成员无法复用上下文，只能重复提问或重复试错。

2. **知识沉淀断裂**
   - 重要决策、方案讨论、代码建议、执行过程散落在不同工具里。
   - 新成员加入后很难快速理解“为什么这么做”。

3. **协作过程不可见**
   - 团队不知道谁正在和 AI 讨论什么、当前进展到哪一步、哪些建议被采纳。
   - AI 参与流程缺乏透明性，导致结果难复盘、难接手。

4. **从“讨论”到“执行”缺少闭环**
   - AI 给出建议后，通常无法自然转成任务、负责人和截止时间。
   - 团队无法把对话直接变成协作执行流。

### 2.3 业务目标

结合现有文档，第一阶段建议聚焦以下业务目标：

- 验证“**共享 AI 会话 + 协同任务跟进**”是否具备明确痛点和使用频率。
- 让一个 3-20 人团队能够在真实项目中持续使用，而不是一次性试用。
- 形成核心留存闭环：
  - 创建共享会话
  - 邀请成员
  - 在会话中提问 AI
  - 从对话生成任务
  - 后续成员继续接力推进
- 为后续知识库、IDE 插件、开放 API、企业版能力打基础，但不在 MVP 阶段全部实现。

### 2.4 产品成功标准

第一阶段更适合关注产品验证，而不是过早追求复杂商业化能力。建议优先关注：

- 团队是否愿意把 AI 对话迁移到共享空间
- 被邀请成员是否会查看并继续跟进会话
- AI 对话是否能实际转化为任务与行动
- 一个会话是否能在多天内持续被复用，而不是“一次性聊天记录”

可作为后续量化指标的方向：

- 工作区创建数 / 周活跃工作区数
- 会话邀请转化率
- 会话被二次跟进的比例
- 从消息转任务的转化率
- 7 天留存 / 30 天留存

## 3. 目标用户与角色

### 3.1 目标用户分层

#### 一级目标用户：中小型研发团队

- 规模：3-20 人，扩展可覆盖到 50 人
- 特征：日常使用 Cursor、Copilot、Claude、ChatGPT 等 AI 工具
- 典型痛点：
  - AI 讨论发生在个人电脑和个人账号里
  - 新成员接手项目成本高
  - 多人同时推进需求、代码、文档时上下文不统一

#### 二级目标用户：初创公司技术团队

- 特征：节奏快，角色边界不固定，AI 参与需求、研发、测试、文档等多个环节
- 核心诉求：
  - 快速保留决策上下文
  - 降低人员切换和信息损耗
  - 把 AI 使用过程纳入项目推进

#### 三级目标用户：远程或异步协作团队

- 特征：时区分散、同步沟通成本高
- 核心诉求：
  - 让未在线成员也能看到完整上下文
  - 减少重复同步和重复讨论

### 3.2 产品角色模型

建议采用“两层角色”设计：**工作区角色** + **会话角色**。

#### 工作区角色

1. **Owner / 所有者**
   - 创建工作区
   - 管理成员、计费、默认配置
   - 拥有最高权限

2. **Admin / 管理员**
   - 管理工作区成员与会话
   - 配置 AI 提供方、策略、权限

3. **Member / 成员**
   - 创建和参与会话
   - 发送消息、创建任务、跟进协作

4. **Viewer / 观察者**
   - 只读访问
   - 适合管理者、顾问、外部协作方

#### 会话角色

1. **Session Admin**
   - 管理会话成员和会话设置
2. **Contributor**
   - 发送消息、触发 AI、创建和更新任务
3. **Observer**
   - 查看会话和任务，但不主动修改

### 3.3 关键角色需求映射

- **Owner/Admin**
  - 需要看到团队整体协作情况
  - 需要管理成员和访问范围
  - 需要保证会话可持续复用

- **Contributor**
  - 需要快速开始共享会话
  - 需要与 AI 实时对话并让团队同步看到结果
  - 需要把对话结果转为任务继续推进

- **Observer**
  - 需要低成本理解背景和当前进度
  - 需要看到关键结论，而不是介入所有细节

## 4. 核心场景梳理

### 场景 1：团队发起一个共享 AI 会话

**触发点：** 某位成员在处理需求分析、接口设计、代码重构、问题排查时，希望团队共同查看与跟进。  
**目标：** 把个人 AI 对话变成可共享的协作入口。  
**关键动作：**

- 创建工作区中的新会话
- 设置名称、描述、可见性
- 邀请团队成员加入
- 成员进入后可查看既有上下文

### 场景 2：多人实时围观并参与 AI 讨论

**触发点：** 团队正在一起推进某个任务，如 API 设计、Bug 排查、方案选择。  
**目标：** 让 AI 对话像团队聊天一样实时共享，而不是个人黑盒操作。  
**关键动作：**

- 成员发送文本、代码块、引用消息
- 某成员触发 AI 生成回复
- AI 回复以流式或分段方式实时同步给在线成员
- 其他成员补充问题、纠偏、确认结论

### 场景 3：从对话直接生成任务并分配

**触发点：** AI 给出了下一步建议、待办项或可执行方案。  
**目标：** 让“讨论”自然流入“执行”。  
**关键动作：**

- 从某条消息一键创建任务
- 自动带入标题、描述、来源消息
- 指定负责人、优先级、截止时间
- 后续在任务上评论与状态流转

### 场景 4：成员中途加入后快速补上下文

**触发点：** 新同事、跨职能成员或晚加入的参与者需要接手任务。  
**目标：** 用最短时间理解来龙去脉。  
**关键动作：**

- 查看会话历史
- 按时间、消息类型、成员过滤
- 从关键任务反查来源消息
- 找到最终方案和未完成事项

### 场景 5：管理者查看当前协作进展

**触发点：** 负责人需要知道团队是否真正借助 AI 提升协作效率。  
**目标：** 快速知道谁在推进什么，哪些事项已采纳、已完成、被阻塞。  
**关键动作：**

- 查看会话列表与活跃度
- 查看任务状态与负责人
- 识别逾期或无人认领事项

## 5. MVP 边界定义

### 5.1 MVP 核心判断

MVP 的目标不是做一个完整的“AI 协作操作系统”，而是优先验证下面这条闭环是否成立：

**共享会话创建 -> 邀请成员 -> 团队共同查看 AI 对话 -> 从对话生成任务 -> 持续跟进**

只要这一闭环可用、可复用、可持续，产品就具备继续扩展的基础。

### 5.2 MVP 必做能力（P0）

1. **基础账号与工作区**
   - 邮箱注册/登录
   - 创建工作区
   - 邀请成员加入工作区

2. **共享会话管理**
   - 创建会话
   - 设置会话名称、描述、可见性
   - 邀请成员加入会话

3. **实时消息系统**
   - 发送文本消息
   - 支持 Markdown / 代码块
   - WebSocket 实时同步
   - 历史消息加载

4. **AI 对话能力**
   - 至少接入一个 AI 提供方
   - 在会话中触发 AI 回答
   - 支持流式输出或分段输出
   - 标记 AI 消息类型与模型信息

5. **任务闭环**
   - 从消息创建任务
   - 任务状态流转：待办 / 进行中 / 已完成 / 已取消
   - 指派负责人、优先级、截止时间
   - 在会话或任务列表中查看任务

6. **最小通知能力**
   - @ 提及通知
   - 任务分配通知

### 5.3 MVP 应做但可后置到 P1 的能力

- 会话内基础关键词搜索
- 消息编辑与删除
- 任务评论
- 会话归档
- 附件上传
- 更细粒度的权限管理

### 5.4 明确不纳入 MVP 的能力

以下能力建议放到 Beta 或 V1.0 之后：

- **IDE 插件（VS Code / JetBrains）**
- **开放 API / Webhook / SDK**
- **知识库归档体系**
- **高级分析看板 / 燃尽图 / 成员工作量统计**
- **企业计费、SSO、2FA、私有化部署**
- **多区域部署、Kubernetes、复杂多服务拆分**
- **多模型路由、复杂 AI 策略编排**
- **展示 AI 原始思维链 / 原始隐藏 prompt**

### 5.5 MVP 成功验收标准

- 一个团队可以在不依赖外部沟通工具的情况下完成至少一次真实协作流程
- 在线成员能在可接受时间内看到同一会话的最新消息和 AI 回复
- 至少可以从对话中稳定地创建并追踪任务
- 晚加入成员能通过历史记录快速理解上下文

## 7. 非功能需求

### 7.1 性能要求

MVP 阶段建议采用**可达成、可验证**的目标，而不是提前承诺超大规模能力。

- 消息实时同步：**P95 < 1 秒**
- AI 首个可见响应分片：**P95 < 3 秒**（不含模型本身极端慢响应）
- 首屏加载时间：**< 2.5 秒**
- 历史消息翻页：**P95 < 800ms**
- 单会话在线协作容量：**50-100 人**

### 7.2 可用性与可靠性

- MVP 服务可用性目标：**99.5%+**
- 客户端断线自动重连
- 消息去重与幂等处理
- AI 调用失败需可重试并可见失败状态
- 关键变更（任务状态、成员加入、权限调整）需有审计记录

### 7.3 安全要求

- 全站 HTTPS
- 密码哈希存储（建议 Argon2 或 bcrypt）
- AI Key 加密存储
- 基于角色的访问控制（RBAC）
- 所有会话与工作区访问必须做服务端权限校验
- 审计日志记录关键操作

### 7.4 可维护性

- 前后端接口需有明确类型定义
- 数据表迁移可追踪、可回滚
- 核心模块边界清晰：认证、工作区、会话、消息、AI、任务、通知
- 所有关键链路具备日志与错误上报

### 7.5 可扩展性

- MVP 先满足单区域部署
- 应用层保持无状态，方便后续水平扩容
- WebSocket 支持后续接入 Redis Adapter
- 搜索优先使用 PostgreSQL Full-Text Search，避免 MVP 引入 Elasticsearch

### 7.6 合规与边界

- 明确区分用户内容、AI 生成内容、系统日志
- 后续如服务企业客户，再补充数据保留策略、导出能力和合规审查
- 不在 MVP 中承诺企业级安全认证（如 SOC2、私有化）交付，但应在设计上预留演进空间

## 8. 技术架构建议

### 8.1 总体建议：模块化单体优先

现阶段最合适的架构不是微服务，而是：

**Web 前端 + 模块化后端单体 + PostgreSQL + Redis（按需）**

原因：

- 当前核心是验证产品闭环，不是解决极端规模问题
- 业务模块边界已经清晰，但流量与团队规模还不足以支撑微服务拆分收益
- 模块化单体可以保留未来拆分可能，同时显著降低开发和运维复杂度

### 8.2 建议技术选型

#### 前端

- `React 18 + TypeScript + Vite`
- 状态管理：`Zustand`
- 服务端状态获取：建议引入 `TanStack Query`
- UI：`Tailwind CSS + shadcn/ui`
- Markdown 渲染 + 代码高亮
- WebSocket 客户端：`socket.io-client`

#### 后端

- `Node.js 20 LTS`
- `Fastify`
- `Socket.IO`
- ORM：建议 `Prisma` 或 `Drizzle` 二选一
- AI 网关：自建 provider 抽象层

#### 数据与基础设施

- 主数据库：`PostgreSQL 15`
- 缓存 / WebSocket 扩展：`Redis`
- 文件存储：对象存储（仅在附件功能落地时引入）
- 搜索：MVP 使用 PostgreSQL 全文检索
- 部署：Docker Compose + 托管数据库优先

### 8.3 推荐模块划分

后端建议按以下模块组织：

1. **Auth**
   - 注册登录
   - Token 管理
   - 密码与身份校验

2. **Workspace**
   - 工作区 CRUD
   - 成员与角色

3. **Session**
   - 会话 CRUD
   - 会话成员
   - 可见性与加入逻辑

4. **Message**
   - 消息持久化
   - 历史消息查询
   - WebSocket 广播

5. **AI Gateway**
   - AI Provider 抽象
   - 流式输出
   - 配置与错误处理

6. **Task**
   - 从消息创建任务
   - 状态流转
   - 任务查询与更新

7. **Notification**
   - @ 提及
   - 任务相关提醒

### 8.4 WebSocket 设计建议

WebSocket 只承担“实时事件分发”，核心写入仍以数据库为准。

建议原则：

- 先写数据库，再广播事件
- 客户端基于消息 ID 去重
- 断线重连后，按最近游标补拉历史消息
- AI 流式响应按 chunk 事件分发，最终落地为一条完整 AI 消息

### 8.5 AI 集成建议

MVP 阶段建议：

- 只接一个主 provider
- 保留统一接口，方便后续扩展多 provider
- 记录但不暴露敏感 AI 调用内部细节
- AI 配置建议支持：
  - 工作区默认配置
  - 会话级可选覆盖（可作为后续能力）

### 8.6 架构上应主动避免的事项

以下事项不建议在 MVP 就投入：

- 微服务拆分
- Elasticsearch 集群
- Kubernetes
- 复杂事件总线或独立 MQ 集群
- IDE 插件与主产品并行开发
- 多模型编排平台化

## 9. 核心数据模型

以下数据模型用于支撑 MVP 与后续扩展，优先列出核心实体。

### 9.1 MVP 核心实体

| 实体 | 说明 | 关键字段 |
|------|------|----------|
| `User` | 系统用户 | `id` `email` `name` `avatar_url` `plan` |
| `Workspace` | 团队组织单元 | `id` `owner_id` `name` `description` `plan` |
| `WorkspaceMember` | 工作区成员关系 | `workspace_id` `user_id` `role` `invited_by` `joined_at` |
| `Session` | 共享 AI 会话 | `id` `workspace_id` `creator_id` `name` `description` `visibility` `is_archived` |
| `SessionMember` | 会话成员关系 | `session_id` `user_id` `role` `last_read_at` |
| `Message` | 会话中的消息 | `id` `session_id` `user_id` `type` `content` `metadata` `timestamp` |
| `MessageMention` | 消息提及关系 | `message_id` `mentioned_user_id` |
| `Task` | 从对话中沉淀的待办事项 | `id` `session_id` `title` `description` `status` `priority` `assignee_id` `source_message_id` |
| `TaskComment` | 任务评论与讨论 | `id` `task_id` `user_id` `content` `parent_id` |
| `SessionAIConfig` | 会话 AI 配置 | `session_id` `provider` `encrypted_key` `last_used_at` |
| `Notification` | 用户通知 | `id` `user_id` `type` `title` `content` `related_type` `related_id` `is_read` |

### 9.2 建议保留但可后置的实体

| 实体 | 用途 |
|------|------|
| `Attachment` | 文件或图片附件 |
| `KnowledgeItem` | 知识库条目 |
| `RefreshToken` | 长会话登录态管理 |
| `AuditLog` | 安全审计与关键操作回溯 |

### 9.3 核心关系

- 一个 `User` 可以拥有多个 `Workspace`
- 一个 `Workspace` 通过 `WorkspaceMember` 关联多个 `User`
- 一个 `Workspace` 下有多个 `Session`
- 一个 `Session` 通过 `SessionMember` 关联多个成员
- 一个 `Session` 下有多条 `Message`
- 一条 `Message` 可以提及多个成员，也可以关联零个或多个附件
- 一个 `Task` 必须属于某个 `Session`，可来源于某条 `Message`
- 一个 `Task` 可以有多个 `TaskComment`
- 一个 `Session` 至少有零或一个有效的 AI 配置
- 一个 `User` 可以收到多条 `Notification`

### 9.4 关键建模建议

1. **工作区与会话分层必须保留**
   - 工作区解决团队组织和权限问题
   - 会话解决具体讨论与协作问题

2. **任务必须可回溯到来源消息**
   - 这是产品形成“从讨论到执行闭环”的关键

3. **AI 消息与普通消息统一建模**
   - 使用 `type` 和 `metadata` 区分
   - 避免维护两套消息系统

4. **权限关系用关联表表达，不写死在用户表**
   - 方便未来支持更细粒度权限

5. **知识库作为后续派生实体，不应干扰 MVP 主链路**
   - MVP 先保证消息和任务链路稳定

## 10. 第一阶段建议结论

基于现有文档，第一阶段建议正式确认以下方向：

1. **产品核心不是“另一个 AI 聊天工具”，而是“团队共享 AI 协作空间”**
2. **第一优先级是验证共享会话与任务协同闭环**
3. **MVP 只做 Web 端，不同时推进 IDE 插件、开放 API 与企业功能**
4. **架构采用模块化单体，避免过度设计**
5. **技术上先满足可用与可验证，再逐步追求复杂扩展能力**

## 11. 待确认问题

这些问题建议作为下一轮需求澄清的输入：

1. **MVP 的 AI 使用模式**
   - 平台统一代调 AI，还是 BYOK（用户自带 Key）优先？
   - 是工作区级配置，还是会话级配置为主？

2. **会话可见性策略**
   - 是否保留 `public / invite_only / private` 三档，还是先简化成 `private / invite_only`？

3. **任务在 MVP 中的呈现方式**
   - 先只做列表足够，还是必须包含看板？

4. **搜索是否进入 MVP**
   - 若要控制范围，建议只做会话内关键词搜索

5. **通知策略**
   - 是否只做站内通知，邮件通知延后？

6. **目标首批用户**
   - 是优先面向研发团队，还是更泛化到所有团队协作场景？
