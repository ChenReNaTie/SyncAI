# SyncAI / 灵悉 AI - 需求收敛说明

**更新时间：** 2026-04-14  
**用途：** 作为当前版本的主需求基线，后续架构设计、界面设计、接入方案、开发拆解均以本文件为准。

---

## 一、产品定位

SyncAI 不是一个新的 AI 编程工具，也不重做 Claude Code 或 Codex。

SyncAI 的定位是：**面向小团队的 AI 编程协作与共享平台**。

它复用底层 AI 编程能力：
- Claude Code
- Codex

它负责的上层能力是：
- 共享 AI 会话历史
- 同步中间过程和最终结果
- 让团队成员在同一个会话里继续发消息
- 从对话中提取轻量 todo
- 搜索和复用历史上下文

一句话概括：

**把原本只存在于个人终端里的 AI 编程过程，转化为团队可共享、可追踪、可搜索、可协作的项目资产。**

---

## 二、目标用户与核心痛点

### 目标用户
- 小团队协作

### 第一核心痛点
- AI 对话历史不能共享

---

## 三、MVP 核心结构

第一版采用以下层级：

- 团队
  - 项目
    - 会话

说明：
- 项目层保留，但做轻量，不做复杂项目管理。
- 一个团队可以有多个项目。
- 一个项目可以有多个会话。

---

## 四、会话规则

### 1. 会话类型
- 共享会话
- 私有会话

### 2. 共享会话
- 项目内所有成员都能看到共享会话历史。
- 项目内所有成员都可以继续发言和追问 AI。
- 新加入项目的成员，默认可以看到该项目下过去的共享会话历史。

### 3. 私有会话
- 私有会话挂在项目下面。
- 私有会话只有创建者自己可见。

### 4. 会话转换
- 私有会话可以转为共享会话。
- 共享会话也可以转回私有会话。
- 但要在产品规则中明确：如果共享期间已有成员看过历史，转回私有只影响后续可见性，不代表历史从未暴露。

### 5. 会话标题与列表
- 会话标题由用户手动填写。
- 会话列表第一版显示：
  - 会话标题
  - 类型（共享 / 私有）
  - 最后更新时间
- 第一版支持：
  - 置顶
  - 收藏

---

## 五、团队与项目规则

### 1. 团队创建与加入
- 用户先创建团队，再手动邀请成员。

### 2. 团队角色
第一版只区分两种角色：
- 管理员
- 普通成员

### 3. 项目成员规则
- 团队成员默认都能进入所有项目。
- 第一版不做复杂的项目成员权限隔离。

### 4. 项目归档
- 第一版支持项目归档。

---

## 六、Todo 规则（MVP）

### 1. 生成方式
- 手动从某条消息中提取为待办。
- 第一版不做 AI 自动建议待办。

### 2. 最小字段
- 标题
- 状态

### 3. 状态集合
第一版只保留两个状态：
- 待办
- 已完成

### 4. 来源关系
- 每个 todo 都要保留来源消息。
- 点击 todo 可以跳回原消息。

### 5. 展示位置
- 第一版放在当前会话的侧边栏 / 侧区。

---

## 七、Claude Code / Codex 集成要求

### 1. 底层 Agent 支持
- 第一版同时支持 Claude Code 和 Codex。
- 但单个会话一次只绑定一个底层 agent。

### 2. Agent / 模型入口
- 第一版只有一个默认入口。
- 第一版不做多模型切换 UI。

### 3. 团队成员发消息规则
- 项目内所有成员都可以往同一个共享会话里继续发消息。

### 4. 消息进入底层 Agent 的方式
- 第一版采用追加到同一个会话继续运行的方式。
- 底层实现仍需有队列 / 串行调度能力，以处理多人连续发消息的情况。

### 5. 平台展示内容原则
用户要求：

**Claude Code / Codex 显示哪些内容结构，SyncAI 页面上也展示对应的内容结构；但样式、颜色、布局由 SyncAI 自己设计。**

第一版至少展示：
- 最终回复
- 关键步骤日志
- 命令执行过程
- 状态变化

这意味着：
- 要做“内容结构一致”
- 不要求“终端样式完全一致”

### 6. 运行状态
第一版至少展示以下状态：
- 正在思考
- 执行中
- 已完成
- 出错

### 7. 消息编辑 / 删除
- 第一版不支持消息编辑
- 第一版不支持消息删除

---

## 八、搜索能力

### 搜索范围
- 第一版需要做项目级 / 全局搜索

### 搜索结果最少显示字段
- 命中的消息内容
- 所属会话
- 所属项目
- 命中人
- 时间

---

## 九、MVP 明确不做

第一版明确不做：
- IDE 插件
- 企业权限体系
- 复杂审批流
- 多模型编排

---

## 十、建议的统一事件模型

为了同时兼容 Claude Code 与 Codex，平台内部建议统一成以下事件类型：
- Session 事件
- Message 事件
- Status 事件
- Tool / Command 事件
- File Change 事件
- Todo 事件

这样做的作用：
- 统一前端展示
- 支持搜索和存档
- 支持 todo 提取
- 支持协作回放
- 支持多 Agent 兼容

---

## 十一、当前结论

SyncAI 第一版的本质不是“再做一个 AI 编程器”，而是：

**在 Claude Code / Codex 之上，做一个面向小团队的共享协作层。**

核心价值有四个：
- 共享 AI 编程历史
- 展示 AI 编程过程
- 支持多人围绕同一会话继续协作
- 让历史内容可搜索、可追溯、可提取 todo
