## 5. 心跳推进标准

后续 heartbeat 不应只做“有没有变化”的被动巡检，而应默认按下面规则推进：

1. 先读取本文件，确认当前阶段与优先级。
2. 再检查 `shared-ai-chat` 当前长期 Codex 会话是否处于正常项目任务态。
3. 如果 Codex 会话不存在、已失败、停在占位提示、停在泛化提示、或长时间没有真实项目输出：
   - 视为需要纠偏；
   - 直接按 `codex-long-session` 的正路重拉或重喂一个**单点具体任务**。
4. 每次 heartbeat 默认只推进**一个最小单点任务**，不要让 Codex 自行大范围摸索。
5. 每次 heartbeat 推进顺序默认按以下优先级选择最近的一项：
   - `project archive` 边界收口
   - `search / replay / todo / visibility` 一致性收口
   - `auth / teams / workspace` 主链路收口
   - integration / e2e 真实链路补齐
   - README / 正式文档口径回收
6. 只有出现以下情况才主动提醒拿铁：
   - 有新的真实产物、测试结果、阶段变化
   - 出现真实阻塞、会话失联、需要拍板
7. 如果没有新的真实进展，就保持静默，不用为了“报平安”打扰。

## 6. 当前 heartbeat 默认下一步

如果下一次 heartbeat 检查时，Codex 仍然停在无意义占位提示或没有真实项目输出，默认直接纠偏到这条任务：

- **只围绕 `project archive` 相关合同测试与 `workspace` 路由实现继续收口**
- 不扩展到无关模块
- 必须带验收标准与最小必要测试

这条规则优先于泛泛的“继续推进项目”。
